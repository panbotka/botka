The task execution output (live and historical) currently shows raw text. It should be rendered like the chat UI — with formatted text, collapsible tool call panels, syntax-highlighted code blocks, etc.

## Current behavior
- Task output is displayed as plain/raw text in the task detail page
- Tool calls, markdown, code blocks — everything is just unformatted text
- Hard to follow what the agent is doing

## Expected behavior
- Task output rendered similarly to the chat view:
  - **Tool calls** shown as collapsible panels (like ToolCallPanel in chat) — tool name, input, output
  - **Markdown** rendered properly (headers, bold, lists, code blocks)
  - **Code blocks** with syntax highlighting (reuse the CodeBlock component from chat)
  - **Text** formatted as readable paragraphs
- Both live streaming output and historical (completed task) output should be rendered this way

## Requirements

### Backend — Structured output parsing
- The task executor already captures Claude Code's NDJSON output (same format as chat streaming)
- Currently the output is likely stored or streamed as raw text
- Parse the NDJSON events into structured data: `assistant` text blocks, `tool_use` events, `tool_result` events, `result` events
- For live streaming: send structured SSE events (similar to chat streaming) instead of raw text lines
- For historical output: store structured data (or parse raw NDJSON on retrieval)

### Backend — Output event types
- Reuse the same event parsing from `internal/claude/runner.go` that chat uses
- Stream events to the frontend as typed SSE events:
  - `text` — assistant text content (markdown)
  - `tool_use` — tool call start (name, input)
  - `tool_result` — tool call result (output)
  - `error` — error events
  - `result` — completion event

### Frontend — Reuse chat rendering components
- Reuse or adapt components from the chat UI:
  - `ToolCallPanel` — for tool calls (collapsible, shows name + input + output)
  - `CodeBlock` — for syntax-highlighted code blocks within markdown
  - Markdown renderer (whatever chat uses — likely react-markdown or similar)
- Create a `TaskOutputView` component that takes the structured events and renders them like a chat conversation (but single-sided — only assistant messages)
- Tool calls default to collapsed (task output can have hundreds of tool calls)

### Frontend — Live streaming
- For running tasks, the output streams via SSE
- Parse the SSE events and append to the rendered view in real-time
- Auto-scroll to bottom as new content arrives (with a "scroll to bottom" button if user scrolled up)

### Frontend — Historical output
- For completed/failed tasks, load the output from the API
- Render the same way as live output, just not streaming

### Tool call summary
- Show a small summary line for each tool call when collapsed:
  - `Read: /path/to/file.go` (show file path for Read)
  - `Edit: /path/to/file.go` (show file path for Edit)
  - `Bash: make test` (show command for Bash)
  - `Grep: "pattern"` (show pattern for Grep)
  - `Write: /path/to/file.go` (show file path for Write)
  - Other tools: just the tool name

### Performance
- Task output can be very large (thousands of events for long tasks)
- Use virtualization or lazy rendering if needed — don't render 10,000 DOM nodes at once
- Or paginate: show last N events with "Load more" at the top

## Implementation Notes
- The chat UI components in `frontend/src/components/` are the reference — reuse as much as possible
- The NDJSON event format is the same for both chat and tasks (Claude Code outputs the same format)
- The task executor in `internal/runner/executor.go` already reads Claude Code's stdout — it just needs to parse and forward structured events instead of raw lines
- Check how the chat SSE handler (`chat.go`) parses events — the task SSE handler should do the same
- Existing tests must pass (`make check`)
