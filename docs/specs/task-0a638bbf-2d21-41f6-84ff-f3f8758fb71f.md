## Problem

The session pool (`internal/claude/pool.go`) pre-warms Claude Code processes to save startup latency, but it does NOT achieve true session reuse. Each message:

1. `PreWarm()` spawns a process with `--resume`, writes a keepalive byte to stdin
2. `RunFromPool()` writes the prompt to stdin, then **closes stdin** (pool.go:229)
3. Process reads until EOF, streams response, and **exits**
4. After response, a **new** process is spawned via `PreWarm()`

This means every message kills and respawns the subprocess. The pool saves ~100-500ms of process startup latency but does NOT reduce token usage. Each new process with `--resume` still reloads the full session context.

## Goal

Investigate whether Claude Code CLI supports keeping a process alive across multiple messages (stdin stays open, multiple prompts sent sequentially). If it does:

1. Refactor `RunFromPool()` to NOT close stdin after writing the prompt
2. Instead, keep the process alive and send subsequent messages to the same stdin
3. Track process lifecycle properly — only kill/respawn when session changes (model change, thread switch, etc.)

If Claude Code CLI does NOT support multiple prompts on a single stdin (i.e., it exits after one response):

1. Document this limitation clearly in code comments
2. Consider whether the pool is worth keeping for just startup latency savings
3. If keeping it, add a comment explaining that it's a latency optimization only, not a token optimization

## Investigation Steps

1. Test whether `claude --resume $ID -p "hello"` can receive multiple prompts on stdin without closing it
2. Check if there's a delimiter or protocol for sending multiple messages
3. Look at Claude Code CLI documentation or `claude --help` for interactive/pipe modes

## Safety
