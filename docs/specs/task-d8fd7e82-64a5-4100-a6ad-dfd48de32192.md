## Bug

Task runner starts new tasks even when 5-hour API usage is above the threshold (e.g. 96% with threshold at 90%).

## Root Cause

Race condition between `UsageMonitor.Start()` and `Runner.RestoreState()` in `cmd/server/main.go` (lines 96-110):

1. `usageMon.Start()` spawns a goroutine that calls `Poll()` asynchronously
2. Main thread immediately continues to `taskRunner.RestoreState()` which can start the scheduler loop
3. First `tick()` calls `IsRateLimited()` before the first `Poll()` completes
4. `m.info` is zero-valued (`FiveHourPct = 0.0`) → `0.0 > 0.90 = false` → task is NOT blocked

## Fix

Make the first usage poll **synchronous** before the scheduler can start. Two changes needed:

### 1. Add synchronous initial poll to `UsageMonitor`

In `internal/runner/usage.go`, add a method or modify `Start()` so the first `Poll()` runs synchronously (on the calling goroutine) before returning. The background goroutine should only handle subsequent periodic polls.

Simplest approach — call `Poll()` before `Start()` in main.go:

```go
// cmd/server/main.go — after creating usageMon
usageMon.Poll()            // Synchronous first poll — blocks until usage data is available
usageMon.Start(context.Background())  // Background polling for updates
```

This ensures `m.info` has real data before `RestoreState()` can start the scheduler.

### 2. Handle "no data" state safely in `IsRateLimited()`

Even with the synchronous first poll, `Poll()` can fail (command error, parse error). In that case `m.info` stays zero-valued and tasks would still bypass the check.

Add a `lastPollOK` bool field to `UsageMonitor`. Set it to `true` only after a successful poll. In `IsRateLimited()`, if `!m.lastPollOK`, return `(true, "no usage data available")` — block tasks when we have no data rather than allowing them.

```go
// In UsageMonitor struct:
lastPollOK bool

// In Poll():
info, err := m.fetchUsage()
if err != nil {
    slog.Error("usage monitor: fetch failed", "error", err)
    m.mu.Lock()
    m.lastPollOK = false
    m.mu.Unlock()
    return
}
m.mu.Lock()
m.info = *info
m.lastPollOK = true
m.mu.Unlock()

// In IsRateLimited():
if !m.lastPollOK {
    return true, "no usage data available yet"
}
```

### 3. Remove the duplicate `Poll()` call from the goroutine in `Start()`

Since we now call `Poll()` synchronously before `Start()`, remove line 60 (`m.Poll()`) from the goroutine in `Start()` to avoid a redundant immediate poll.
