Allow the user to change the maximum number of concurrent task workers from the Settings page. Currently hardcoded via `MAX_WORKERS` env var (default 2). Store in DB so it persists and can be changed at runtime without restart.

## Current state

**Backend:**
- `MAX_WORKERS` env var → `config.Config.MaxWorkers` (int, default 2)
- `Runner` reads `r.config.MaxWorkers` in the scheduling loop (`runner.go:291`): `len(r.executors) >= r.config.MaxWorkers`
- `Status` struct exposes `max_workers` in runner status API
- No server-side settings table exists

**Frontend:**
- Settings page (`SettingsPage.tsx`) has sections for Personas, Tags, Memories, and client-side preferences (theme, font size, send on enter)
- Client-side settings are in localStorage via `SettingsContext`
- No server-side settings management

## Requirements

### Backend — Server settings table
- Create a migration adding an `app_settings` table:
  ```sql
  CREATE TABLE app_settings (
    key VARCHAR(100) PRIMARY KEY,
    value TEXT NOT NULL,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
  );
  INSERT INTO app_settings (key, value) VALUES ('max_workers', '2');
  ```
- This is a simple key-value table for server-side settings that need to persist in DB

### Backend — Settings model and helpers
- Add a `Setting` model in `internal/models/setting.go` (or similar)
- Add helper functions: `GetSetting(db, key) string`, `SetSetting(db, key, value)`
- These just read/write from the `app_settings` table

### Backend — Settings API endpoints
- `GET /api/v1/settings` — returns all server-side settings as a JSON object: `{"data": {"max_workers": 2}}`
- `PUT /api/v1/settings` — accepts partial update: `{"max_workers": 3}` → updates the DB
- Validate `max_workers`: must be integer 1-10

### Backend — Runner reads from DB
- On startup, Runner reads `max_workers` from `app_settings` table (falling back to `config.MaxWorkers` env var if not set)
- When the setting is updated via API, notify the Runner to reload (simplest: Runner re-reads from DB on each scheduling tick, or the handler calls a `SetMaxWorkers` method on the Runner)
- Add `SetMaxWorkers(n int)` method on Runner that updates the in-memory value (thread-safe with mutex)

### Backend — Draining state
- When the number of active executors exceeds `maxWorkers` (e.g. user lowered the limit while tasks are running), the Runner is in a "draining" state
- Do NOT kill running tasks — just stop picking new ones until active count drops below the limit
- Expose this in the `Status` struct: add `Draining bool` field, true when `len(r.executors) > maxWorkers`
- The `max_workers` field in Status should reflect the NEW limit, so the frontend can show "5/2 workers (draining)"

### Frontend — Dashboard draining indicator
- When `runnerStatus.draining` is true, show a visual indicator on the dashboard runner status area
- Display something like "5/2 workers (draining)" — active count / max workers with a "draining" badge
- Use amber/yellow color for the draining state
- Once active count drops to or below max workers, the indicator disappears

### Frontend — Settings page section
- Add a "Task Runner" section to `SettingsPage.tsx`
- Show "Max Workers" with a number input or stepper (1-10)
- On change, call `PUT /api/v1/settings` with the new value
- Show current value from `GET /api/v1/settings` on page load
- Display a brief explanation: "Maximum concurrent task execution slots"

### Frontend — API client
- Add `fetchServerSettings()` and `updateServerSettings(settings)` to `api/client.ts`

## Implementation Notes
- The `app_settings` table is intentionally generic (key-value) so future settings can be added without migrations
- The Runner's `config.MaxWorkers` becomes the initial default — DB value overrides it
- The scheduling loop check at `runner.go:291` already uses `r.config.MaxWorkers` — either update config in place or add a separate field that gets checked
- Simplest approach: add `maxWorkers int` field on Runner (separate from config), initialize from DB on start, update via `SetMaxWorkers()`
- Existing tests must pass (`make check`)
