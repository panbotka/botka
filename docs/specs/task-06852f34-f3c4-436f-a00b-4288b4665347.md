# MCP Config Generation & Subprocess Integration

Generate temporary `.mcp.json` files with resolved MCP servers and pass them to Claude Code subprocesses via the `--mcp-config` flag. This is the core integration that makes managed MCP servers actually work in chat and task sessions.

## Context

Botka spawns Claude Code subprocesses in two paths:
1. **Chat sessions** — `internal/claude/pool.go` (SessionManager, persistent streams) and `internal/claude/runner.go` (one-shot Run function)
2. **Task execution** — `internal/runner/executor.go` (batch one-shot)

Neither currently passes any MCP flags. MCP servers are stored in the `mcp_servers` database table. A helper function `ResolveMCPServers(db, threadID, projectID)` returns the list of active servers for a given context (created by a previous task).

## Requirements

### MCP Config File Generation

Create `internal/claude/mcp_config.go` with a function:

```go
func GenerateMCPConfig(servers []models.MCPServer) (string, error)
```

This function:
1. Takes a list of resolved MCPServer models
2. Builds the Claude Code `.mcp.json` format:
   ```json
   {
     "mcpServers": {
       "server-name": {
         "command": "...", "args": [...], "env": {...}
       },
       "another-server": {
         "type": "sse", "url": "...", "headers": {...}
       }
     }
   }
   ```
3. For stdio servers, the config JSON contains `command` (string), `args` ([]string), `env` (map[string]string). Map these to the `.mcp.json` format directly.
4. For sse servers, the config JSON contains `url` (string), `headers` (map[string]string). Output `type`, `url`, and `headers` fields.
5. Writes to a temp file at `{CLAUDE_CONTEXT_DIR}/mcp-{thread-id|task-id}.json`
6. Returns the absolute file path
7. If no servers are resolved, return empty string (no file needed, no flag to pass)

### Chat Session Integration

**In `internal/claude/pool.go` — `buildStreamArgs()` function:**
- Accept an optional MCP config file path parameter
- If non-empty, append `"--mcp-config", path` to the args slice
- The caller (SessionManager) must resolve MCP servers before spawning

**In `internal/claude/pool.go` — session spawning logic:**
- Before creating a new process, call `ResolveMCPServers(db, &threadID, projectID)` 
- Then call `GenerateMCPConfig(servers)` to get the temp file path
- Pass the path into the args builder
- The DB reference is available through the handler chain — pass it into StreamConfig or SessionManager as needed

**In `internal/claude/runner.go` — `Run()` function (one-shot fallback):**
- Same pattern: resolve servers, generate config, add --mcp-config to args

**Important:** When a session is reused from the pool (same model, same session ID), the MCP config was already set at spawn time. If the user changes MCP servers on a thread, the session must be invalidated (killed and respawned). Add MCP server list hash to the session cache key or invalidation check — similar to how model changes already invalidate sessions.

### Task Execution Integration

**In `internal/runner/executor.go` — `executeTask()` method:**
- Before spawning claude, resolve MCP servers: `ResolveMCPServers(db, nil, &task.ProjectID)`
- Generate config file: `GenerateMCPConfig(servers)`
- Add `"--mcp-config", path` to the command args (before the `-p` flag)
- Clean up the temp file after task execution completes (defer)

### Cleanup

- Chat session config files: cleaned up when the session is killed or evicted
- Task config files: cleaned up via defer after task execution
- On Botka startup, clean any stale `mcp-*.json` files in the context directory

## Testing

- Test GenerateMCPConfig with stdio server, sse server, mixed, empty list
- Test that generated JSON matches Claude Code's expected format
- Test config file is created and contains valid JSON
- Test args builder includes --mcp-config when path is non-empty, omits when empty
- Test session invalidation when MCP servers change
