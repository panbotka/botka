## How the dark mode system works

The app uses Tailwind CSS 4 with a custom variant defined in `frontend/src/index.css`:
```css
@custom-variant dark (&:where(.dark, .dark *, .dark-green, .dark-green *, .dark-blue, .dark-blue *));
```

Three dark themes (dark, dark-green, dark-blue) override CSS variables in `:root.dark`, `:root.dark-green`, `:root.dark-blue`. The theme class is set on `<html>` by `SettingsContext.tsx`.

Some color values ARE remapped via CSS variables (e.g. `--color-zinc-50: #18181b` in dark), so classes like `bg-zinc-50` automatically adapt. But hardcoded values like `bg-white`, explicit light colors, and native `<select>`/`<input>` elements need explicit `dark:` overrides.

## Implementation approach

1. Start with a systematic grep for `bg-white`, `bg-zinc-50`, `bg-zinc-100`, `border-zinc-200`, `border-zinc-300`, `text-zinc-900`, `text-zinc-700` in `frontend/src/` to find all instances missing `dark:` counterparts
2. Fix the Login page first (most visible — it has zero dark mode support)
3. Fix all Settings tab form inputs and buttons
4. Fix overlay components (CommandPalette, ModelPicker, SearchOverlay)
5. Fix remaining inline components (MessageBubble file cards, ChatInput, etc.)
6. Test all three dark themes (dark, dark-green, dark-blue) — they should all benefit from the same fixes since they share the custom variant
