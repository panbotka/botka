# Add pagination to tasks list page

The tasks list page loads all tasks client-side and filters locally. With 200+ tasks this is slow and the list is too long. Add classic server-side pagination.

## Requirements

- Add a pagination component at the bottom of the task list showing page numbers (1, 2, 3...) with previous/next buttons
- 100 tasks per page
- Switch from client-side filtering to server-side: send `status`, `project_id`, `limit`, and `offset` query params to the backend
- The backend already supports `limit`, `offset`, `status`, and `project_id` params in `GET /api/v1/tasks` — no backend changes should be needed
- When the user changes status filter or project filter, reset to page 1
- Persist current page in URL search params (e.g. `?status=pending&page=2`) so browser back/forward works
- The status tab counts (All: N, Pending: N, etc.) must still reflect the total counts across all pages — use the existing `GET /api/v1/tasks/stats` endpoint or the `total` field from the list response
- Update `useTasks` hook to accept `limit` and `offset` params and pass them to `fetchTasks`

## Implementation Notes

- The `taskFilter` function in `internal/handlers/task.go` already handles `status` and `project_id` filtering, and `parsePagination` handles `limit`/`offset` — the backend is ready
- The `fetchTasks` client function already accepts `limit` and `offset` params
- Status tab counts currently come from iterating the loaded tasks client-side — this needs to change since we no longer load all tasks. Use the `total` from the API response for the active filter, and consider a separate lightweight request for the counts, or use the existing `/tasks/stats` endpoint
- Keep the current sort order: `priority DESC, created_at ASC` (done tasks: `completed_at DESC`)
