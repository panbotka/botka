When the user switches away from the browser tab/window and comes back, automatically refresh stale data (tasks, projects, threads) so changes made externally (e.g., via CLI) are visible without a manual page reload.

## Problem

Data hooks (`useTasks`, `useProjects`, and thread loading) only fetch on mount or manual action. When the user creates tasks via the CLI or another tool and switches back to the Botka web UI, the data is stale and requires a full page refresh.

## Solution

Use the `visibilitychange` DOM event to detect when the tab becomes visible again, and trigger refetches. Add a small staleness threshold (e.g., 2 seconds since last fetch) to avoid redundant requests when rapidly switching tabs.

### Option A: Central hook (preferred)

Create a reusable hook `frontend/src/hooks/useRefreshOnFocus.ts`:

```typescript
import { useEffect, useRef } from 'react'

/**
 * Calls `onRefresh` when the browser tab becomes visible again,
 * but only if at least `minStaleMs` have elapsed since the last refresh.
 */
export function useRefreshOnFocus(onRefresh: () => void, minStaleMs = 2000) {
  const lastRefresh = useRef(Date.now())

  useEffect(() => {
    // Track when data was last refreshed
    lastRefresh.current = Date.now()
  }, [onRefresh])

  useEffect(() => {
    const handler = () => {
      if (document.visibilityState === 'visible' && Date.now() - lastRefresh.current > minStaleMs) {
        lastRefresh.current = Date.now()
        onRefresh()
      }
    }
    document.addEventListener('visibilitychange', handler)
    return () => document.removeEventListener('visibilitychange', handler)
  }, [onRefresh, minStaleMs])
}
```

### Option B: Integrate into existing hooks

Alternatively, add `visibilitychange` handling directly into:
- `frontend/src/hooks/useTasks.ts` ŌĆö call `refetch()` on tab focus
- `frontend/src/hooks/useProjects.ts` ŌĆö call `refetch()` on tab focus
- Thread list loading in `frontend/src/pages/ChatPage.tsx` or wherever threads are fetched

Option A is cleaner ŌĆö create the hook once, then wire it into each consumer.

### Where to wire it

1. **Tasks page** (`frontend/src/pages/TasksPage.tsx`):
   - Uses `useTasks()` which returns `refetch`
   - Add: `useRefreshOnFocus(refetch)`

2. **Task detail page** (`frontend/src/pages/TaskDetailPage.tsx`):
   - Has its own task fetch ŌĆö add refresh on focus

3. **Chat page** (`frontend/src/pages/ChatPage.tsx`):
   - Thread list loading ŌĆö refetch threads on focus
   - Read the code to find the thread fetch/refresh function

4. **Projects page** (`frontend/src/pages/ProjectsPage.tsx`):
   - Uses `useProjects()` which returns `refetch`
   - Add: `useRefreshOnFocus(refetch)`

5. **Dashboard page** (`frontend/src/pages/DashboardPage.tsx`):
   - Likely has its own data fetching ŌĆö add refresh on focus

6. **Runner status** (`frontend/src/components/RunnerStatus.tsx`):
   - If it shows runner state, refresh on focus too

### Important considerations

- Use `useCallback` for the refetch functions passed to `useRefreshOnFocus` to avoid unnecessary listener re-registrations
- The existing `refetch` functions from `useTasks` and `useProjects` are already wrapped in `useCallback`, so they should be stable
- Don't show loading spinners during background refresh ŌĆö only update the data silently. If the hooks currently set `loading = true` during refetch, consider adding a `silent` parameter or a separate `isRefreshing` state
- The staleness threshold prevents rapid re-fetches if the user is quickly tabbing between windows
