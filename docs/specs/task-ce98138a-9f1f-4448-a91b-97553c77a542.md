## Task 5: Task Handler and Runner Engine

Port Saiduler's task management API and runner engine to Botka.

### Context
This is the core task scheduling functionality from Saiduler. Tasks are created, queued, and executed by the runner which spawns Claude Code processes.

### Reference Files
- `/home/pi/projects/saiduler/internal/handlers/task.go` — Task CRUD handlers
- `/home/pi/projects/saiduler/internal/handlers/runner.go` — Runner control handlers
- `/home/pi/projects/saiduler/internal/handlers/output.go` — SSE task output streaming
- `/home/pi/projects/saiduler/internal/runner/runner.go` — Scheduler loop
- `/home/pi/projects/saiduler/internal/runner/executor.go` — Task execution (spawn claude, classify, verify, PR)
- `/home/pi/projects/saiduler/internal/runner/parser.go` — stream-json parser
- `/home/pi/projects/saiduler/internal/runner/buffer.go` — Ring buffer for live output
- `/home/pi/projects/saiduler/internal/runner/usage.go` — Anthropic API usage monitor

### What to Create

1. **`internal/runner/`** — Port entire package from Saiduler:
   - `runner.go` — Runner struct, NewRunner, Start, Pause, Stop, Shutdown, Status, tick, pickTask, launchTask
   - `executor.go` — Executor, Execute, syncSpec, setupBranch, buildPrompt, spawnClaude, classifyOutcome, maybeVerify, pushAndCreatePR
   - `parser.go` — ParseStream, Event types for batch mode parsing
   - `buffer.go` — Ring buffer for live task output with subscribers
   - `usage.go` — UsageMonitor reading Claude credentials for API rate limits

   Adjust import paths from `github.com/saiduler/saiduler` to `botka`. Keep the executor's own spawn logic (with SysProcAttr, Setpgid, WaitDelay) — it has different lifecycle needs than the chat subprocess runner.

2. **`internal/handlers/task.go`** — Port from Saiduler:
   - `GET /api/v1/tasks` — List with status/project_id filters, pagination
   - `POST /api/v1/tasks` — Create task
   - `POST /api/v1/tasks/batch-status` — Batch update statuses
   - `POST /api/v1/tasks/reorder` — Batch update priorities
   - `GET /api/v1/tasks/:id` — Get with execution history
   - `PUT /api/v1/tasks/:id` — Update task
   - `DELETE /api/v1/tasks/:id` — Cancel or hard-delete
   - `POST /api/v1/tasks/:id/retry` — Retry failed tasks

3. **`internal/handlers/runner.go`** — Port from Saiduler:
   - `GET /api/v1/runner/status` — Runner state + active tasks + usage info
   - `POST /api/v1/runner/start` — Start (optional count for task limit)
   - `POST /api/v1/runner/pause` — Pause
   - `POST /api/v1/runner/stop` — Hard stop
   - `POST /api/v1/runner/usage/refresh` — Refresh usage

4. **`internal/handlers/output.go`** — Port SSE task output streaming:
   - `GET /api/v1/tasks/:id/output` — Server-Sent Events for live task output

5. **Wire everything** in main.go:
   - Create UsageMonitor, start it
   - Create Runner with config
   - Restore runner state on startup
   - Register all routes
   - Shutdown runner on exit

### Verification
- Create a task via `POST /api/v1/tasks`
- Get runner status via `GET /api/v1/runner/status`
- Start runner via `POST /api/v1/runner/start`
- Observe task execution via SSE output stream
- Task transitions through statuses correctly
