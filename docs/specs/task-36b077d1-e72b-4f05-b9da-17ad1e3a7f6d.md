# Custom Context Field Per Chat Thread

Add a free-form text field to each chat thread that gets injected into the context at the start of every chat session. This lets the user paste arbitrary reference material (API specs, DB schemas, notes, etc.) that the AI always has available for that thread.

## Database / Model

- Add a new column to the `threads` table via migration:
  - `custom_context TEXT NOT NULL DEFAULT ''`
- Add to the Thread model in `internal/models/thread.go`:
  - `CustomContext string` with `gorm:"type:text;not null;default:''" json:"custom_context"`

## Context Assembly

In `internal/claude/context.go`, add a new layer for custom context. Insert it between the thread system prompt (layer 6) and the project CLAUDE.md (layer 7):

- If `customContext` is non-empty, append it as: `# Reference Context\n\n<content>`
- Pass the custom_context value through from wherever `AssembleContext` is called — check the function signature and the call sites to add the parameter

## Backend API

The existing thread update endpoint (`PUT /api/v1/threads/:id` or similar) should already accept arbitrary fields. Check how other thread fields like `system_prompt` are updated and add `custom_context` to the same flow:

- Accept `custom_context` in the thread update request body
- No length validation needed beyond what the DB allows (TEXT type)
- Return the field in thread GET responses

## Frontend — Thread Settings UI

Add a "Custom Context" textarea to the thread settings/edit area. Check where `system_prompt` is currently edited for the thread — the custom context editor should be placed nearby.

- Label: "Custom Context" with helper text: "Reference material included in every message. Paste API docs, schemas, notes, etc."
- Use a `<textarea>` with reasonable size (at least 6 rows, resizable)
- Auto-save on blur (same pattern as system_prompt or other thread fields)
- Show character count or a subtle indicator of content length
- Place it in the thread sidebar or thread settings panel — wherever system_prompt is edited

## MCP Tools

Add two MCP tools to `internal/mcp/server.go`:

### `get_thread_context`

**Parameters:**
- `thread_id` (integer, required)

**Behavior:**
- Return the thread's custom_context content

### `set_thread_context`

**Parameters:**
- `thread_id` (integer, required)
- `content` (string, required) — the full custom context text (replaces existing)

**Behavior:**
- Update the thread's custom_context field
- Return success with the updated content length

These tools allow setting context programmatically, e.g. "paste this API spec into thread 42's context".

## Edge Cases

- Empty custom_context (default) — no context layer added, no UI indication needed beyond the empty textarea
- Very large content — no hard limit, but the textarea should handle large text gracefully (scrollable)
- The custom_context is distinct from system_prompt: system_prompt contains instructions/persona for the AI, custom_context contains reference data/information. Both are injected into context but serve different purposes and should be clearly separated in the UI.
