When Claude Code returns a transient error like "Error in input stream", automatically retry by evicting the session and resending the message, instead of just showing the error.

## Context
"Error in input stream" is a transient error from Claude Code's `--input-format stream-json` parser. It happens occasionally when the internal session state becomes inconsistent. The fix on our side is simple: kill the session and try again with a fresh one.

## Requirements

### Backend — Auto-retry in streamResponse
- In `chat.go` `streamEventsToClient`, when a result event has `is_error: true` and the error message matches known transient errors:
  - "Error in input stream"
  - Empty error string (`""`)
- Evict the current session (`Sessions.Evict(threadID)`)
- Retry `streamResponse` once (with a fresh session)
- Only retry once — if the retry also fails, show the error normally
- Log the retry: `[chat] thread %d: transient error "%s", retrying with fresh session`

### Transient error detection
- Create a helper function `isTransientError(msg string) bool` in `chat.go` or `runner.go`
- Match known transient patterns (case-insensitive):
  - "error in input stream"
  - empty string
- Keep the list small and explicit — only add errors we're confident are transient

### Edge cases
- Don't retry rate limit errors, "No conversation found" errors, or other non-transient errors
- The "No conversation found" retry is already handled separately (stale session fallback)
- Don't retry if the user already received partial response content (only retry if no content was streamed yet, or if the error came as the very first event)
- Protect against infinite retry loops — use a `retryCount` parameter, max 1 retry

## Implementation Notes
- The retry should be transparent to the user — they just see the response arrive (slightly delayed)
- If retry succeeds, no error is shown at all
- If retry fails, show the error normally (via the inline error block or banner)
- `Sessions.Evict(threadID)` already exists and cleanly kills the session
- The `streamResponse` function already has a fallback path (`isFallback`) — the retry is different: it creates a new persistent session, not a one-shot Run
- Existing tests must pass (`make check`)

## Safety
**NEVER run `make deploy`, `make install-service`, `systemctl restart botka`, or any command that would restart the Botka service.**
