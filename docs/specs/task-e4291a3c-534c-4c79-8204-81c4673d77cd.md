# Cron Jobs — Frontend Page

Add a new "Cron Jobs" page to the Botka frontend for managing scheduled AI prompts with execution history.

## Context

The backend API already exists:
- `GET /api/v1/cron-jobs` — list all
- `GET /api/v1/cron-jobs/:id` — get one
- `POST /api/v1/cron-jobs` — create
- `PATCH /api/v1/cron-jobs/:id` — update
- `DELETE /api/v1/cron-jobs/:id` — delete
- `GET /api/v1/cron-jobs/:id/executions` — execution history (paginated)
- `POST /api/v1/cron-jobs/:id/run` — manual trigger

Each cron job: `{id, name, schedule, prompt, project_id, project, enabled, timeout_minutes, model, last_run_at, last_status, created_at}`
Each execution: `{id, cron_job_id, status, output, error_message, cost_usd, input_tokens, output_tokens, duration_ms, started_at, finished_at}`

## Requirements

### Navigation

Add "Cron Jobs" to the main sidebar navigation (use `Clock` or `Timer` icon from Lucide). Place it near the Tasks link since they're conceptually related.

### Page Layout

Create `frontend/src/pages/CronJobsPage.tsx` with two views:

**List View (default):**
- Header: "Cron Jobs" with an "Add" button
- List/card layout showing all cron jobs
- Each card shows:
  - Name (bold)
  - Schedule in human-readable form AND raw cron expression (e.g. "Every Monday at 9:00 — `0 9 * * 1`"). Use a helper function to convert common cron patterns to readable text. For complex patterns just show the expression.
  - Project name (as a badge/link)
  - Enabled toggle (inline, immediate PATCH on change)
  - Last run: relative time ("2 hours ago") + status badge
  - Status badge colors: green for "success", red for "failed", orange for "timeout", gray for never run
- Empty state: "No cron jobs configured. Schedule your first AI task."

**Detail View (click on a cron job):**
- Can be a modal, slide-over, or separate route — match existing patterns in the app (check how task detail works)
- Show all cron job fields (editable)
- Execution history below, ordered newest first
- Each execution row: started_at, duration, status badge, cost, token count
- Click on execution to expand and show full output (collapsible, monospace/markdown rendered)
- "Run Now" button to manually trigger
- "Delete" button with confirmation

### Add/Edit Form

Modal or inline form (match existing patterns like Settings > Personas):

- **Name** — text input, required
- **Schedule** — text input for 5-field cron expression, required. Show validation feedback ("Invalid cron expression" or parsed human-readable preview like "Runs every day at 09:00")
- **Project** — dropdown/select, required. Fetch from GET /api/v1/projects
- **Prompt** — textarea (large, resizable), required. This is the AI prompt.
- **Model** — optional dropdown (sonnet/opus/haiku or empty for default)
- **Timeout** — number input, minutes, default 30
- **Enabled** — checkbox, default true

### Cron Expression Helper

Add a small helper that converts cron expressions to readable text for common patterns:
- `0 9 * * *` → "Every day at 09:00"
- `0 9 * * 1` → "Every Monday at 09:00"
- `*/30 * * * *` → "Every 30 minutes"
- `0 0 1 * *` → "1st of every month at 00:00"
- For anything complex, just show the raw expression

Consider using a lightweight library like `cronstrue` (npm) if available, or write a simple helper for the most common patterns.

### API Client

Add to `frontend/src/api/client.ts`:
- `listCronJobs()`
- `getCronJob(id)`
- `createCronJob(data)`
- `updateCronJob(id, data)`
- `deleteCronJob(id)`
- `listCronExecutions(jobId, { limit, offset })`
- `triggerCronJob(id)`

### Route

Add route in the router (check `App.tsx` or wherever routes are defined):
- `/cron-jobs` — CronJobsPage

## UI Details

- Follow existing page patterns (look at TasksPage for layout reference)
- Use Tailwind CSS consistent with the rest of the app
- Status badges: small colored pills — green "success", red "failed", orange "timeout", gray "never run"
- Execution output: render in a collapsible panel with monospace font, similar to how task output is displayed
- The prompt textarea should be generously sized — prompts can be long
- Enabled toggle should update immediately without needing to save the whole form
- Show relative times with title attribute for absolute time (existing pattern in the app)

## Edge Cases

- Cron job with no executions yet — show "Never run" status
- Very long prompt text — truncate in list view, show full in detail
- Very long execution output — lazy load or paginate
- Project deleted — cron job CASCADE deleted, handle gracefully in UI
- Manual trigger while already running — show feedback ("Already running")
