In PWA standalone mode (installed to home screen on iPhone), the top edge of the UI is too close to the status bar / Dynamic Island area. The back arrow button in the chat view is not tappable because it's hidden behind the system UI.

## Context
- The bottom nav safe area issue is tracked separately (viewport-fit=cover fix) — this task covers the **top edge**
- In PWA standalone mode there's no Safari toolbar, so the app content starts right at the top of the screen
- On iPhones with Dynamic Island or notch, the top area is occupied by system UI
- The back button (arrow left) in the chat thread header is positioned too high and gets obscured

## Requirements

### Verify viewport-fit=cover
- `viewport-fit=cover` must be set in the viewport meta tag (may already be done by the bottom nav fix)
- Without it, `env(safe-area-inset-top)` returns 0 and no padding is applied

### Add top safe area padding
- Apply `padding-top: env(safe-area-inset-top, 0px)` to the top-level app shell or the header/navbar area
- This should affect:
  - The chat thread header (where the back button lives)
  - Any other page headers that have interactive elements at the very top
  - The main app header/top bar if there is one
- The padding should only add space in PWA mode (on regular browsers with address bars, `safe-area-inset-top` is 0 so it's a no-op)

### Check all pages
- Chat page: back button must be tappable
- Thread list: header area
- Tasks page: header area
- Settings, Help, Dashboard: verify top elements are reachable
- Any page with a top-positioned action button

### Styling
- The safe area padding should feel natural — match the background color of the header
- Don't add excessive padding on devices without notch/Dynamic Island
- Follow existing design language

## Implementation Notes
- The fix is likely adding `pt-[env(safe-area-inset-top)]` or equivalent to the top-level layout wrapper or individual page headers
- In Tailwind, arbitrary values work: `pt-[env(safe-area-inset-top,0px)]`
- Alternatively, apply it via a CSS class in the global stylesheet
- Check `App.tsx` or the layout wrapper component for the best place to apply this globally
- The `apple-mobile-web-app-status-bar-style` is set to `black-translucent` which means the status bar overlaps the app content — this is expected, but the app must account for it with safe-area-inset-top
- Existing tests must pass (`make check`)

## Safety
**NEVER run `make deploy`, `make install-service`, `systemctl restart botka`, or any command that would restart the Botka service.**
