Add support for external users who can only access specific chat threads assigned to them. They cannot create threads, access tasks, settings, or any admin features. Depends on the auth system being implemented first (task: "Add authentication — single user with password + passkeys").

## Prerequisite
This task MUST be implemented AFTER the main authentication task is done. It extends the auth system with roles and thread-level access control.

## Requirements

### Database — User roles and thread access
- Add `role` column to `users` table (migration): `admin` (default, the main user) or `external`
- Create a `thread_access` table (migration):
  - `id` (bigserial)
  - `user_id` (bigint, FK to users)
  - `thread_id` (bigint, FK to threads)
  - `created_at`
  - Unique constraint on (user_id, thread_id)
- External users can only see and interact with threads they have explicit access to

### Backend — External user management (admin only)
- `POST /api/v1/users` — create external user `{"username": "...", "password": "..."}` (admin only)
- `GET /api/v1/users` — list all users (admin only)
- `DELETE /api/v1/users/:id` — delete external user (admin only, cannot delete self)
- `PUT /api/v1/users/:id/password` — reset external user's password (admin only)
- `POST /api/v1/users/:id/threads` — grant thread access `{"thread_id": N}` (admin only)
- `DELETE /api/v1/users/:id/threads/:thread_id` — revoke thread access (admin only)
- `GET /api/v1/users/:id/threads` — list threads an external user can access (admin only)

### Backend — Authorization middleware
- Extend the existing auth middleware with role-based checks:
  - **Admin**: full access to everything (current behavior)
  - **External**: restricted access:
    - `GET /api/v1/threads` — only returns threads they have access to
    - `GET /api/v1/threads/:id` — 403 if not assigned
    - `POST /api/v1/threads/:id/messages` — allowed only for assigned threads (can send messages and chat)
    - `GET /api/v1/threads/:id/messages` — allowed only for assigned threads
    - All other endpoints: 403 (tasks, settings, projects, personas, tags, cost, users, etc.)
- External users CANNOT:
  - Create or delete threads
  - Access tasks, projects, settings, cost dashboard
  - See other threads
  - Manage personas, tags, memories
  - Access MCP endpoints

### Backend — Thread list filtering
- The threads list endpoint must filter by user role:
  - Admin: sees all threads
  - External: sees only threads from `thread_access` table
- Thread detail endpoint: check access before returning

### Frontend — External user experience
- External users see a simplified UI:
  - Only the Chat page in bottom nav (no Dashboard, Tasks, Cost, Settings)
  - Thread list shows only their assigned threads
  - Full chat functionality within assigned threads (read, write, voice input if enabled)
  - No thread creation button
  - No thread deletion
  - No system prompt or sources editing
- Help page: accessible (but content could be simplified — optional)

### Frontend — Admin user management
- Add a "Users" section in Settings (or a separate page accessible from Settings):
  - List external users (username, created date, number of assigned threads)
  - "Create user" form (username + password)
  - Per-user: manage thread access (list of assigned threads with add/remove)
  - Delete user button
  - Reset password button

### Login page
- Same login page for both admin and external users
- After login, redirect based on role:
  - Admin: Dashboard (/)
  - External: Chat page (/chat)
- The login endpoint response should include the user's role

## Implementation Notes
- The auth middleware from the prerequisite task handles session cookies — extend it with role checking
- Use Gin middleware chaining: auth middleware (checks session) → role middleware (checks permissions)
- For thread access checks, a simple DB query: `SELECT 1 FROM thread_access WHERE user_id = ? AND thread_id = ?`
- Consider caching thread access per session to avoid DB hits on every request
- External users use username + password only (no passkey support needed initially)
- Existing tests must pass (`make check`)
- Write tests for the authorization logic — especially ensuring external users can't access unauthorized resources
