## Problem

When the user sends a message and navigates away from the chat page (e.g. to Settings), the ChatView component unmounts and the SSE connection for streaming the assistant response is closed. When the user returns to the chat, the response is lost — no message appears.

The Claude subprocess on the backend continues running and finishes, but the frontend never receives the streamed events because the SSE listener was destroyed on navigation.

## Requirements

1. **Assistant responses must not be lost** when navigating away from chat and back.
2. When the user returns to the chat thread, they should see the complete assistant response (or the in-progress stream if still running).

## Implementation approach

The simplest fix: **when ChatView mounts and detects an active process for the current thread, reconnect to the SSE stream.**

### Option A: Reconnect on mount (simplest)

In `ChatView.tsx`, on mount (or when `threadId` changes):
1. Check if there's an active process for this thread (via the `/processes` endpoint or the `useProcesses` hook)
2. If yes, reconnect to the SSE stream endpoint to pick up the ongoing response
3. The backend SSE endpoint should support late-joining — if the process is still running, the client picks up from where the stream currently is

Check how the SSE streaming endpoint works in the backend (`internal/handlers/chat.go`). If it supports reconnection mid-stream, this is straightforward.

### Option B: Refetch messages on mount (fallback)

If the SSE stream can't be rejoined mid-flight:
1. On ChatView mount, always fetch the latest messages from the API
2. This ensures any completed responses are shown even if the stream was missed
3. For in-progress streams, also reconnect to SSE to catch the remainder

### Option C: Move SSE listener out of ChatView

Move the SSE connection management to a higher level (e.g. `ChatPage` or a global context) so it persists across route changes. This is more invasive but the most robust solution.

### Recommendation

Start with Option B (refetch messages on mount) as it's the simplest and handles the most common case (user comes back after response is done). Then add SSE reconnection for in-progress streams if needed.

Currently, ChatView likely already fetches messages on mount. The issue might be that:
- Messages are fetched but the just-completed response isn't in the DB yet (timing)
- Or messages are fetched from local state that was cleared on unmount

Check the message loading logic in ChatView and the `useMessages` or similar hook to understand the exact failure mode.

## Files to check

- `frontend/src/components/ChatView.tsx` — SSE connection lifecycle, message loading on mount
- `frontend/src/hooks/` — any hooks managing SSE streams or message fetching
- `internal/handlers/chat.go` — SSE streaming endpoint, whether it supports reconnection

## CRITICAL: Do NOT deploy or restart

**NEVER run `make deploy`, `make install-service`, `systemctl restart`, or any command that would restart the Botka service.** You are running inside Botka — deploying would kill your own process and leave the task stuck in "running" state forever.
