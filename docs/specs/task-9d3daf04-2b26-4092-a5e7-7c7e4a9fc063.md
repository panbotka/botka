## Problem

The "Search messages" box in the thread sidebar doesn't work well. It uses PostgreSQL full-text search with English stemming (`plainto_tsquery('english', ?)` + `tsvector`), which:
1. Doesn't do fuzzy/substring matching
2. Doesn't handle Czech text or diacritics properly
3. Fails on partial words

The user wants functional fuzzy search that ignores diacritics (e.g. searching "recept" should match "Recept" and "řečept", searching "zapis" should match "zápiš").

## Requirements

1. **Diacritic-insensitive search** — "recept" matches "řečept", "zapis" matches "zápiš"  
2. **Substring/fuzzy matching** — partial words should match
3. **Case-insensitive**
4. Keep the existing UX: debounced search in the sidebar, results grouped by thread with snippets

## Implementation approach

Replace the full-text search query in `internal/handlers/search.go` with a simpler `ILIKE`-based search using PostgreSQL's `unaccent` extension.

### Step 1: Create a new migration

Create a migration that enables the `unaccent` extension:

```sql
CREATE EXTENSION IF NOT EXISTS unaccent;
```

### Step 2: Update search query in `internal/handlers/search.go`

Replace the `ts_rank` / `plainto_tsquery` query (~lines 65-75) with something like:

```sql
SELECT
    m.id, m.role, m.created_at,
    t.id, t.title, t.model, t.created_at, t.updated_at,
    -- Generate snippet: extract substring around match
    substring(m.content FROM greatest(1, position(lower(unaccent(?)) in lower(unaccent(m.content))) - 50) FOR 120) AS snippet
FROM messages m
JOIN threads t ON t.id = m.thread_id
WHERE unaccent(lower(m.content)) LIKE '%' || unaccent(lower(?)) || '%'
ORDER BY m.created_at DESC
LIMIT 50
```

Or use `ILIKE` with `unaccent()`. The key point is:
- `unaccent()` strips diacritics from both the query and content
- `ILIKE` or `lower()` handles case insensitivity
- Substring matching with `%query%` pattern

### Step 3: Snippet generation

The current `ts_headline()` won't work anymore. Generate snippets either:
- In SQL: extract ~100 chars around the match position
- In Go: after fetching the full content, find the match position and extract surrounding text

Keep the `<mark>` tags around the matched text for highlighting.

### Step 4: Consider performance

For the current data size this should be fine. If needed later, add a GIN trigram index:
```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_messages_content_trgm ON messages USING gin(unaccent(lower(content)) gin_trgm_ops);
```

But don't add this yet — keep it simple.
