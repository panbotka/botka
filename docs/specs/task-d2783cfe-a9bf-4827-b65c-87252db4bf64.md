# Signal Bridge DB Model, Migration, and API Endpoints

Database schema and API endpoints for linking Botka threads to Signal group chats.

## Requirements

### Database model

Add a new `signal_bridges` table (not a column on threads — keep it separate for clean separation of concerns):

- `id` (bigserial, primary key)
- `thread_id` (int8, foreign key to threads, unique — one bridge per thread)
- `group_id` (text, not null) — Signal group ID (base64 string like `4vnBAIsEAQqiNuELxVNJqfhDzHz4ZwMn9T0BjZtr+ng=`)
- `group_name` (text) — cached group display name
- `active` (boolean, default true) — enable/disable bridge without deleting
- `created_at`, `updated_at` (timestamps)

Create a GORM model in `internal/models/signal_bridge.go` with appropriate struct tags and table name.

### Migration

Create a new SQL migration pair in `migrations/` following the existing numbering convention. Check the latest migration number and increment.

### API Endpoints

Add a new handler file `internal/handlers/signal.go` with these endpoints:

1. **GET /api/v1/signal/groups** — List available Signal groups
   - Calls `signal.Client.ListGroups()` from the signal client package
   - Returns `{"data": [{"id": "...", "name": "...", "member_count": N}]}`

2. **GET /api/v1/threads/:id/signal-bridge** — Get bridge status for a thread
   - Returns the SignalBridge record if exists, or 404
   - Response: `{"data": {"group_id": "...", "group_name": "...", "active": true}}`

3. **PUT /api/v1/threads/:id/signal-bridge** — Create or update bridge
   - Body: `{"group_id": "...", "group_name": "..."}`
   - Creates or updates the SignalBridge record for this thread
   - Returns the created/updated record

4. **DELETE /api/v1/threads/:id/signal-bridge** — Remove bridge
   - Deletes the SignalBridge record
   - Returns 204 No Content

### Route Registration

Register the new endpoints in the existing router setup following the pattern used by other handlers. Signal endpoints should require authentication (same middleware as other API routes).

### Tests

- Handler integration tests following the existing pattern in `internal/handlers/` (httptest + Gin test mode)
- Test CRUD operations on signal bridges
- Test unique constraint (one bridge per thread)
- Test 404 when no bridge exists
