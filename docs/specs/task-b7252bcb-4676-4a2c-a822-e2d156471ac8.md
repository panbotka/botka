# Dev & Deploy Command Buttons in Chat UI

When a chat thread is assigned to a project that has configured dev and/or deploy commands, show quick action buttons in the chat interface so the user can start dev or deploy without navigating to the project detail page.

## Current State

- Project commands (dev/deploy) already work via `POST /api/v1/projects/:id/run-command`, `GET /api/v1/projects/:id/commands`, and `DELETE /api/v1/projects/:id/commands/:pid`
- The `ProjectDetailPage.tsx` already has a full `CommandsSection` component with Start/Stop/Deploy buttons, polling, confirmation dialog, and toast notifications
- API client functions already exist: `runProjectCommand`, `fetchProjectCommands`, `killProjectCommand` in `frontend/src/api/client.ts`
- Threads have an optional `project_id` field linking them to a project

## Requirements

### Button Placement

- Add dev/deploy buttons to the chat view header area (the top bar of `ChatView.tsx` where the thread title, model, and project name are shown)
- Buttons should only appear when the current thread has an assigned project AND that project has at least one command configured (dev_command or deploy_command)
- Keep them compact — small icon buttons or icon+short-label buttons that fit the header aesthetic

### Button States

- **Idle (command not running):** Show a small Play/Rocket icon button. Clicking runs the command.
- **Running:** Show the button as active/highlighted (e.g. green pulse dot or different color) with a Stop icon. Clicking stops the command.
- **No command configured:** Don't show the button at all

### Polling for Status

- When the chat thread has a project with commands, poll `GET /api/v1/projects/:id/commands` every 5 seconds to check running status (same pattern as `CommandsSection` in `ProjectDetailPage.tsx`)
- Stop polling when the thread has no project or the project has no commands configured

### Deploy Confirmation

- Deploy always requires confirmation before execution (window.confirm or a small modal is fine)
- Dev does not require confirmation

### Feedback

- Show a brief toast or inline notification after starting/stopping: "Dev started (PID 12345)" / "Deploy started" / "Process stopped"
- Show errors inline if the command fails to start

### Data Flow

- The `ChatView` component needs access to the current thread's project data (including `dev_command` and `deploy_command` fields). Check how the thread's project is currently loaded — the thread object has `project_id` and likely the project object is available via the threads/projects data already loaded in the app.
- If the project data isn't readily available with command fields, fetch it via the existing project detail API endpoint.

## Implementation Notes

- Reuse the existing API client functions (`runProjectCommand`, `fetchProjectCommands`, `killProjectCommand`) — do not create new ones
- Look at `CommandsSection` in `ProjectDetailPage.tsx` (around line 597) for the exact pattern of polling, running, killing, and toast display — replicate that logic in a compact form
- Consider extracting a small `useProjectCommands(projectId)` hook that encapsulates the polling and state management, so both `ProjectDetailPage` and `ChatView` can use it
- The buttons should be visually consistent with other chat header controls (model picker, etc.)
