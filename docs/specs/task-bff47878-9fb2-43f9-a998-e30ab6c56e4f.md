## Bug

When viewing a running task at `/tasks/:id`, the UI shows "No output available — task may be orphaned" even though the task is actively running with a live Claude Code process. The user sees no output at all.

## Root Cause

In `internal/handlers/output.go` (the `streamTaskOutput` function), the SSE endpoint polls for the executor's output buffer 10 times with 500ms delays (5 seconds total). If the buffer doesn't exist after 5 seconds but the task status is `running` in the DB, it sends an `error` SSE event with the orphaned message and **closes the connection permanently**.

This fails in two scenarios:

1. **Slow buffer creation:** The task status is set to `running` in the DB before the executor goroutine creates the buffer (there's a gap between `pickNextTask()` setting status and `executeTask()` creating the buffer in `runner.go` ~line 510). If the client connects during this window and buffer creation takes >5s (possible under load on RPi), the endpoint gives up.

2. **Page navigation:** User opens the task page well after the task started. The buffer exists but for some reason the 10-poll window isn't enough (e.g., GC pressure, slow DB query for status check).

The frontend `useSSE` hook (`frontend/src/hooks/useSSE.ts`) receives the error event, sets the error state, and renders the error message permanently with no retry.

## Fix

### Backend (`internal/handlers/output.go`)

Change the buffer polling strategy in `streamTaskOutput`:

- **Increase polling duration**: Poll for longer (e.g., 60 attempts × 500ms = 30 seconds) before giving up. A running task's buffer should appear within seconds — 5s is too aggressive for a loaded RPi.
- **Re-check DB status during polling**: If the task transitions away from `running` during the wait (e.g., it completed or failed), stop polling early and send a `done` event instead of the orphaned error.
- **Only send orphaned error as last resort**: After the extended polling, if the buffer still doesn't exist AND the task is still `running` in the DB, then it's genuinely orphaned.

Suggested approach:
```go
// Poll for buffer with periodic DB status checks
maxAttempts := 60
for i := range maxAttempts {
    buf = bp.GetBuffer(taskId)
    if buf != nil {
        break
    }
    // Every 10 attempts, re-check if task is still running
    if i > 0 && i%10 == 0 {
        status, err := sq.QueryTaskStatus(taskId)
        if err != nil || status != models.TaskStatusRunning {
            // Task finished while we were waiting
            setSSEHeaders(c)
            fmt.Fprintf(c.Writer, "event: done\ndata: {}\n\n")
            c.Writer.Flush()
            return
        }
    }
    time.Sleep(500 * time.Millisecond)
}
```

### Frontend (`frontend/src/hooks/useSSE.ts`)

Add retry logic for the `error` event:

- When receiving an "orphaned" error event, don't permanently give up. Instead, retry the SSE connection after a delay (e.g., 3 seconds), up to 5 retries.
- Only show the error message permanently after all retries are exhausted.
- This handles the case where the buffer briefly doesn't exist but appears shortly after.

### Tests

- Add/update tests in `internal/handlers/output_test.go` to verify:
  - Buffer appearing after initial polling attempts still connects successfully
  - Task completing during polling sends `done` event (not orphaned error)
  - Genuinely orphaned task (running in DB, no buffer after full timeout) still shows error
