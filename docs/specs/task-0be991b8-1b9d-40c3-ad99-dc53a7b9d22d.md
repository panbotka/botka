# Fix Passkey Login

Passkey registration works correctly, but logging in with a registered passkey fails. The user can register passkeys in Settings but cannot use them to sign in on the Login page.

## Investigation Guide

The issue is somewhere in the login flow. Here are the key files and the full flow to trace:

### Backend Flow

1. `POST /api/v1/auth/passkey/login/begin` — `internal/handlers/webauthn.go` `LoginBegin()` (line 146)
   - Calls `h.wan.BeginDiscoverableLogin()` which returns a `*protocol.CredentialAssertion`
   - Stores session in memory via `storeSession()`
   - Returns the assertion wrapped in `respondOK()` → `{"data": <assertion>}`
   
2. `POST /api/v1/auth/passkey/login/finish` — `internal/handlers/webauthn.go` `LoginFinish()` (line 158)
   - Finds a login session via `findAnyLoginSession()` — looks for sessions where `UserID` is empty (discoverable flow)
   - Calls `h.wan.FinishPasskeyLogin()` with a user handler callback
   - The callback receives `userHandle` bytes, decodes it as uint64, and looks up the user
   - Creates a session cookie via `h.authHandler.CreateSessionForUser()`

### Frontend Flow

1. `LoginPage.tsx` `handlePasskeyLogin()` (line 66)
   - Calls `passkeyLoginBegin()` from `api/client.ts`
   - The client function (line 766) expects response shape `{ data: { publicKey: ... } }` and extracts `res.data.publicKey`
   - Converts base64url challenge and credential IDs to ArrayBuffers
   - Calls `navigator.credentials.get({ publicKey: publicKeyOptions })`
   
2. After browser authenticator responds:
   - Calls `passkeyLoginFinish(credential)` from `api/client.ts` (line 773)
   - Serializes the `AuthenticatorAssertionResponse` fields to base64url
   - Sends JSON to the backend

### Likely Issues to Check

1. **Response shape mismatch:** The `go-webauthn/webauthn` library's `BeginDiscoverableLogin()` returns a `*protocol.CredentialAssertion` which serializes to JSON with a `publicKey` wrapper. The frontend (line 770) does `res.data.publicKey` to unwrap it. Verify the actual JSON shape matches — if the library wraps it differently or the frontend double-unwraps, the options will be undefined.

2. **Session lookup in LoginFinish:** `findAnyLoginSession()` (line 303) looks for sessions where `len(entry.data.UserID) == 0`. Make sure the login session stored by `BeginDiscoverableLogin` actually has empty/nil UserID. If it doesn't, the session won't be found and login returns "no pending login".

3. **UserHandle encoding mismatch:** During registration, `WebAuthnID()` encodes the user ID as `binary.BigEndian.PutUint64`. During login, `LoginFinish` decodes with `binary.BigEndian.Uint64(userHandle)`. The `userHandle` comes from the authenticator which stores what was set during registration. If the authenticator doesn't return `userHandle` for discoverable credentials, or if it's encoded differently, the user lookup will fail.

4. **Base64url encoding/decoding:** The frontend converts between base64url and ArrayBuffers. Check that the `bufferToBase64url` and `base64urlToBuffer` functions handle padding correctly and match what the go-webauthn library expects.

5. **Auth middleware:** The passkey login endpoints ARE in the public path list (`internal/middleware/auth.go` lines 61-62), so auth middleware should not block them. But verify this is working.
