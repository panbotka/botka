# MCP Server CRUD API Handlers

Add REST API endpoints for managing MCP server configurations. These let users create, list, update, and delete MCP server definitions stored in the `mcp_servers` table.

## Requirements

### Handler File

Create `internal/handlers/mcp_server_handler.go` following the existing handler patterns (see `project_handler.go` or `persona_handler.go` for reference).

### Endpoints

**GET /api/v1/mcp-servers**
- List all MCP servers ordered by name
- Response: `{"data": [...], "total": N}`
- Each item includes all fields: id, name, server_type, config, is_default, active, created_at, updated_at

**POST /api/v1/mcp-servers**
- Create a new MCP server
- Request body: `{"name": "...", "server_type": "stdio|sse", "config": {...}, "is_default": false}`
- Validate: name required and unique, server_type must be "stdio" or "sse", config must be valid JSON
- For stdio config, validate `command` field exists (string). `args` ([]string) and `env` (map[string]string) are optional.
- For sse config, validate `url` field exists (string). `headers` (map[string]string) is optional.
- Response: `{"data": <created server>}` with 201 status

**PATCH /api/v1/mcp-servers/:id**
- Update an existing MCP server (partial update)
- Any subset of: name, server_type, config, is_default, active
- Same validation rules as create
- If name changes, check uniqueness
- Response: `{"data": <updated server>}`

**DELETE /api/v1/mcp-servers/:id**
- Delete an MCP server and all its thread/project associations (CASCADE handles this)
- Response: 204 No Content
- Return 404 if not found

### Route Registration

Register routes in the existing router setup (see how other handlers are registered). All endpoints require authentication (use the existing auth middleware).

### Error Handling

- 400 for validation errors with descriptive message
- 404 for not found
- 409 for duplicate name conflict
- Follow existing error response pattern: `{"error": "message"}`

## Testing

Write handler integration tests in `internal/handlers/mcp_server_handler_test.go` following the existing test patterns (see other handler test files). Use the test database (`botka_test`). Tests should cover:
- List empty, list with data
- Create valid stdio server, create valid SSE server
- Create with missing name (400), duplicate name (409), invalid server_type (400)
- Update name, toggle is_default, toggle active
- Delete existing, delete non-existent (404)
