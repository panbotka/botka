# Add MCP Tools for Project Settings, Command Execution, Threads, and Thread Sources

The MCP server currently only exposes task management tools (create_task, list_tasks, get_task, update_task) and project listing (list_projects). Add tools so project settings can be updated, dev/deploy commands can be executed, threads can be listed, and thread URL sources can be managed — all via MCP without needing the web UI.

## Current MCP Implementation

- `internal/mcp/server.go` — all tool definitions, handlers, and JSON-RPC dispatch
- Tools are registered in `listToolsResult()` with JSON Schema input definitions
- Handlers are mapped in `toolHandlers()` as `map[string]func(json.RawMessage, json.RawMessage) *response`
- Each handler parses arguments from `json.RawMessage`, calls DB operations or runner methods, and returns a JSON-RPC response

## Requirements

### Tool: `update_project`

Update a project's configuration fields.

**Parameters:**
- `project_name` (string, required) — project name (case-insensitive lookup)
- `dev_command` (string, optional) — shell command for dev environment
- `deploy_command` (string, optional) — shell command for production deploy
- `dev_port` (integer, optional) — port the dev server listens on
- `deploy_port` (integer, optional) — port for deploy
- `verification_command` (string, optional) — command to verify task execution
- `branch_strategy` (string, optional) — "main" or "feature_branch"

**Behavior:**
- Look up project by name (case-insensitive), return error if not found
- Only update fields that are present in the request (partial update)
- For string fields: empty string clears the value (sets to NULL), non-empty sets it
- For integer fields: 0 clears the value, positive value sets it
- Validate branch_strategy is one of "main" or "feature_branch" if provided
- Return the updated project as JSON text content

### Tool: `run_command`

Execute a project's configured dev or deploy command.

**Parameters:**
- `project_name` (string, required) — project name (case-insensitive lookup)
- `command` (string, required) — "dev" or "deploy"

**Behavior:**
- Look up project by name, return error if not found
- Check the requested command is configured, return error if not
- Execute using the shared `CommandTracker` instance (see implementation notes)
- Return the PID and command type on success

### Tool: `list_commands`

List running commands for a project.

**Parameters:**
- `project_name` (string, required)

**Behavior:**
- Return list of running commands (pid, command_type, command, started_at, alive)
- Clean up dead processes from tracker

### Tool: `kill_command`

Kill a running command.

**Parameters:**
- `project_name` (string, required)
- `pid` (integer, required)

**Behavior:**
- Kill the process group (same logic as HTTP handler's KillCommand)
- Return success/failure message

### Tool: `list_threads`

List chat threads, optionally filtered by project.

**Parameters:**
- `project_name` (string, optional) — filter threads by project name
- `limit` (integer, optional, default 20) — max results
- `offset` (integer, optional, default 0) — pagination offset

**Behavior:**
- Query threads from DB, ordered by `updated_at DESC`
- If `project_name` is provided, resolve to project ID and filter by `project_id`
- Return array of threads with: id, title, project_id, project name, model, pinned, archived, created_at, updated_at
- Do NOT return message contents (just thread metadata)

### Tool: `list_thread_sources`

List URL sources attached to a thread.

**Parameters:**
- `thread_id` (integer, required)

**Behavior:**
- Query thread sources ordered by position ASC
- Return array with: id, url, label, position

### Tool: `add_thread_source`

Add a URL source to a thread.

**Parameters:**
- `thread_id` (integer, required)
- `url` (string, required)
- `label` (string, optional)

**Behavior:**
- Validate URL is non-empty
- Calculate next position (MAX(position) + 1 for this thread)
- Create the source and return it

### Tool: `remove_thread_source`

Remove a URL source from a thread.

**Parameters:**
- `thread_id` (integer, required)
- `source_id` (integer, required)

**Behavior:**
- Delete the source matching both thread_id and source_id
- Return success/not-found

### Tool: `update_thread_source`

Update a thread source's URL or label.

**Parameters:**
- `thread_id` (integer, required)
- `source_id` (integer, required)
- `url` (string, optional)
- `label` (string, optional)

**Behavior:**
- Look up source by thread_id + source_id
- Update only provided fields
- Return updated source

## Implementation Notes

- Follow the exact pattern of existing tools in `server.go`: define in `listToolsResult()`, add handler in `toolHandlers()`, implement handler method
- The `CommandTracker` from `internal/handlers/command.go` needs to be shared between the HTTP handler and MCP server. Pass the same instance to both.
- Update `NewServer()` to accept `*handlers.CommandTracker` as parameter
- Update the call site in `cmd/server/main.go` to pass the tracker to the MCP server
- The command execution logic (bash -c, SysProcAttr, process reaping) should ideally be a method on CommandTracker itself to avoid duplication. Consider extracting `CommandTracker.Run(project *models.Project, commandType string) (*RunningCommand, error)` that both handlers call.
- For thread sources, use the same DB operations as `internal/handlers/thread_source.go` — query ThreadSource model directly
- For list_threads, use the Thread model from `internal/models/` — preload the Project association to include project name
- Add tests for the new tools following the existing test patterns in `internal/mcp/server_test.go`
