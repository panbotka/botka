Add a stop button that interrupts the current Claude response WITHOUT killing the session. The user wants to stop a response, tweak the prompt, and send again — all within the same session.

## How it works

Claude Code in `--input-format stream-json` mode handles SIGINT (like Ctrl+C in terminal): it stops the current response and emits a result event, but the process stays alive and ready for the next message on stdin.

**Key difference:**
- `cancel()` / `Registry.Kill()` → kills the entire process (session lost)
- `cmd.Process.Signal(os.Interrupt)` → interrupts current response only (session survives)

## Current state

**Backend** (`internal/claude/pool.go`):
- `Session` struct has `cmd *exec.Cmd` with access to `cmd.Process.Pid`
- `SendMessage` goroutine reads events in a loop until `KindResult` — a SIGINT should cause Claude to emit a result event (possibly with error/cancelled flag), ending the loop naturally
- `busy` flag tracks when a message is in-flight

**Backend** (`internal/handlers/process.go`):
- `DELETE /api/v1/processes/:id` calls `Registry.Kill()` which calls `cancel()` — this kills the process entirely (NOT what we want)

**Frontend**:
- No stop button exists. Only `/clear` kills the process
- `isStreaming` state is tracked in `ChatView.tsx`

## Requirements

### Backend — Interrupt method on SessionManager
- Add `Interrupt(threadID int64) error` method to `SessionManager`
- It sends `syscall.SIGINT` to the session's `cmd.Process` (NOT cancel/kill)
- Returns error if no session exists or session is not busy
- The `SendMessage` goroutine should handle this gracefully — Claude Code will emit a result event after receiving SIGINT, so the scanner loop ends naturally
- After interrupt, the session remains alive and `busy` becomes false — ready for the next message

### Backend — Interrupt endpoint
- Add `POST /api/v1/threads/:id/interrupt` endpoint
- Calls `SessionManager.Interrupt(threadID)`
- Returns 200 on success, 404 if no active session, 409 if not currently streaming
- Keep the existing `DELETE /api/v1/processes/:id` (kill) endpoint unchanged

### Frontend — API client
- Add `interruptThread(threadId: number)` to API client
- Calls `POST /api/v1/threads/${threadId}/interrupt`

### Frontend — Stop button
- Show a stop button while Claude is streaming a response (`isStreaming` is true)
- Position: replace or overlay the send button in the input area
- On click: call `interruptThread(threadId)`
- The partial response already received stays visible (don't delete it)
- After Claude stops, the button transitions back to the normal send button
- User can immediately type a new message and send it (same session, no restart)

### Frontend — Button design
- Use Lucide `Square` or `StopCircle` icon
- Prominent but not alarming — fits the existing design language
- Smooth transition between send → stop → send states

### Edge cases
- If interrupt fails (response already finished), ignore the error
- If Claude doesn't respond to SIGINT within ~2 seconds, the session might be stuck — log a warning but don't kill it automatically
- The partial assistant message should be saved as-is
- Test that after interrupting, a new message works correctly on the same session

## Implementation Notes
- `cmd.Process.Signal(syscall.SIGINT)` is the key — this is `os.Interrupt` in Go
- The `SendMessage` scanner loop already breaks on `KindResult` — verify that Claude emits a result event after SIGINT in stream-json mode (if not, may need a timeout fallback)
- Don't touch `Registry.Kill` or `Evict` — those remain for full session teardown
- Existing tests must pass (`make check`)
