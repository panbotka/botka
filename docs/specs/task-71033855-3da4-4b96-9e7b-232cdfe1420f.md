# Configurable Project Commands (Dev & Deploy)

Add two configurable shell commands per project ‚Äî one for starting the development environment and one for deploying to production. Users can configure these in the project settings and execute them from the UI with a single click.

## Requirements

### Database / Model

- Add two new nullable TEXT columns to the `projects` table via a new migration:
  - `dev_command` ‚Äî shell command to start dev environment (e.g. `make frontend-dev`, `docker-compose up`)
  - `deploy_command` ‚Äî shell command to deploy to production (e.g. `make deploy`, `./scripts/deploy.sh`)
- Add corresponding fields to the `Project` model in `internal/models/project.go`:
  - `DevCommand *string` with `gorm:"type:text" json:"dev_command"`
  - `DeployCommand *string` with `gorm:"type:text" json:"deploy_command"`

### Backend API

#### Project Update (`PUT /api/v1/projects/:id`)

- Accept `dev_command` and `deploy_command` in the update request body (same pattern as `verification_command`)
- Validate max length (use the same `maxVerificationCmdLength` constant or a shared `maxCommandLength`)
- Persist to database

#### Command Execution Endpoint

- Add `POST /api/v1/projects/:id/run-command` endpoint
- Request body: `{"command": "dev" | "deploy"}`
- Look up the project, resolve which command to run based on the `command` field
- If the command is not configured (null/empty), return 400 with a clear message like "dev command not configured for this project"
- Execute the command as a shell process: `bash -c <command>` in the project's `Path` directory
- The process should run **detached/in background** ‚Äî the endpoint returns immediately with `{"data": {"pid": 12345, "command_type": "dev"}}`. Do NOT wait for the command to complete.
- Use `exec.Command` with `SysProcAttr.Setpgid = true` so the process runs independently and survives if Botka restarts
- Store running process info in memory (a simple map protected by mutex) so the UI can check status and kill processes

#### Process Status & Kill

- Add `GET /api/v1/projects/:id/commands` ‚Äî returns currently tracked running commands for this project with pid, command_type, started_at, and whether the process is still alive (check with `os.FindProcess` + signal 0)
- Add `DELETE /api/v1/projects/:id/commands/:pid` ‚Äî kills the process (and its process group via negative PID to kill children too). Remove from tracking map.
- Clean up dead processes from the tracking map when status is queried

### Frontend ‚Äî Project Settings

In both `ProjectsPage.tsx` (inline edit) and `ProjectDetailPage.tsx` (detail panel), add input fields for the two new commands alongside the existing `verification_command` field:

- "Dev command" input with placeholder like `make frontend-dev`
- "Deploy command" input with placeholder like `make deploy`
- Same save/dirty pattern as `verification_command` ‚Äî auto-save on blur if changed
- Update the `Project` type in `frontend/src/types/index.ts` to include `dev_command?: string` and `deploy_command?: string`
- Update the API client's project update function to send these fields

### Frontend ‚Äî Command Execution UI

On the `ProjectDetailPage.tsx`, add a "Commands" section (or buttons in the header area) with:

- Two buttons: "Start Dev" (with a Play icon) and "Deploy" (with a Rocket or Upload icon)
- Each button is disabled/hidden if the corresponding command is not configured
- Clicking a button:
  1. Shows a confirmation dialog for Deploy (not needed for Dev)
  2. Calls `POST /api/v1/projects/:id/run-command` with the command type
  3. Shows a brief success toast with the PID: "Dev started (PID 12345)"
- If a command of that type is already running (from the status endpoint), show it as active with a "Stop" button instead
- Periodically poll `GET /api/v1/projects/:id/commands` (every 5 seconds, or on focus) to update running status
- "Stop" button calls `DELETE /api/v1/projects/:id/commands/:pid` and refreshes status

### Discovery Sync

The project discovery sync (`internal/projects/discovery.go`) must preserve these new fields (same as it preserves `verification_command`, `branch_strategy`, etc.) ‚Äî only update `name`, `path`, and `active` during sync, leaving command fields untouched.

## Edge Cases

- If a dev command is already running and user clicks "Start Dev" again, warn them ("Dev is already running. Start another instance?") or block it
- Deploy confirmation should clearly show the command that will be executed
- Process tracking is in-memory only ‚Äî after Botka restart, previously tracked PIDs are lost (this is acceptable, orphaned processes keep running independently)
- Commands run as the same user as Botka (pi). No sandboxing needed.
- Long-running dev processes (like `make frontend-dev`) stay alive until explicitly stopped or the machine reboots

## Implementation Notes

- Look at `internal/runner/executor.go` for patterns on running shell commands with process groups (`Setpgid`)
- The process tracking map could be a simple struct in a new file `internal/handlers/command.go` or added to the project handler
- Keep the execution handler simple ‚Äî no streaming output, no websockets. Just fire-and-forget with PID tracking.
