# Fix Live Output SSE Showing Perpetual RECONNECTING

When viewing a running task's detail page, the live output section shows "RECONNECTING" indefinitely instead of streaming Claude Code output.

## Root Cause

The SSE output handler at `internal/handlers/output.go:42` polls for the task's in-memory buffer for 5 seconds (10 retries × 500ms). If the buffer doesn't exist, it sends a `done` SSE event and closes. The frontend's EventSource receives `done`, then the `onerror` handler triggers a reconnect. This creates an infinite cycle: connect → 5s wait → `done` → reconnect → repeat.

The buffer doesn't exist because:
1. **No executor is running** — the runner may be stopped/paused, or the task's status is `running` in DB but it was orphaned (e.g., from a service restart after `recoverOrphanedTasks` already ran). No executor means no buffer.
2. **Buffer was already cleaned up** — if the task completed between when the page loaded and when EventSource connected, `finishTask()` already closed and deleted the buffer from the map.

## Requirements

### Fix 1: Output handler should check task status in DB before looping

In `internal/handlers/output.go`, the `streamTaskOutput` function should:

1. After the 5-second buffer poll fails (buffer is nil), **query the task's actual status from the database** before sending `done`.
2. If the task status is NOT `running` (i.e., it's `done`, `failed`, `needs_review`, etc.), send the `done` event — this is correct, the task completed.
3. If the task status IS `running` but no buffer exists, this is an orphaned task with no executor. Send a specific error event (e.g., `event: error\ndata: {"message": "No output available — task may be orphaned"}\n\n`) so the frontend can display a helpful message instead of endlessly reconnecting.

The handler needs access to `*gorm.DB` — it's already passed to `RegisterOutputRoute` (line 27) but only used for the raw output handler. Pass it to the SSE handler too.

### Fix 2: Frontend should stop reconnecting on `done` event

In `frontend/src/hooks/useSSE.ts`, the `done` event handler already sets `doneRef.current = true` and closes the EventSource (line 62-74). This should prevent reconnects. But the current behavior shows perpetual reconnecting, which means the `done` event is either not being received or the handler is racing with `onerror`.

**Investigation needed:** The issue might be that the handler sends `done` as `event: done\ndata: {}\n\n` but the EventSource interprets the connection closing as an error BEFORE processing the done event. When an SSE connection closes after sending data, `onerror` may fire before or instead of the named event handler. Check if the frontend receives the `done` event or only gets `onerror`.

Possible fixes:
- Add a small delay after sending the `done` event before the handler returns (e.g., `time.Sleep(100 * time.Millisecond)` after flush)
- In the frontend `onerror`, check if the connection was clean (EventSource.readyState) before reconnecting
- Add an `event: error` listener in the frontend that handles the "no buffer" case gracefully

### Fix 3: Frontend should show appropriate message for orphaned tasks

In `frontend/src/components/LiveOutput.tsx` or `TaskDetailPage.tsx`, if the SSE connection fails repeatedly (done received immediately without data, or error event), show a message like "Output not available" instead of perpetual "RECONNECTING".
