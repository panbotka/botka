# Debug "claude process exited unexpectedly" Errors

The error "claude process exited unexpectedly" occurs frequently during chat sessions. This error originates in `internal/claude/pool.go:330` when the persistent Claude Code subprocess dies mid-conversation — the stdout scanner stops returning lines without a result event, and the process is found to be dead.

## Goal

Investigate why the Claude Code process crashes, improve error reporting so the root cause is visible, and add resilience so the user experience recovers gracefully.

## Investigation Areas

### 1. Better Error Diagnostics

The current error path (pool.go:327-336) checks `s.stderrBuf.String()` for details, but:
- The `stderrBuffer` only holds the last 20 lines — if the process crashes with a long stack trace, the useful part may be truncated
- The exit code is never captured or reported — `cmd.Wait()` runs in a background goroutine (pool.go:227-234) but doesn't store the exit code
- There's no logging of the process exit code or signal that killed it

**Fix:** 
- Store the process exit code/signal in the Session struct when `cmd.Wait()` completes
- Include exit code in the error message: "claude process exited unexpectedly (exit code 1)" or "claude process killed by signal: SIGKILL"
- Increase `maxStderrLines` from 20 to 50 in `runner.go:483`
- Log the full stderr content at ERROR level when the process dies unexpectedly (not just include in the user-facing message)

### 2. Process Exit Monitoring

The background goroutine at pool.go:227-234 marks the session dead when the process exits, but doesn't log WHY:

```go
go func() {
    _ = cmd.Wait()  // exit reason discarded!
    m.mu.Lock()
    defer m.mu.Unlock()
    if !s.dead {
        s.dead = true
        log.Printf("[session] process exited for thread %d (session %s)", threadID, sessionPrefix)
    }
}()
```

**Fix:**
- Capture `cmd.Wait()` error and extract the exit code via `exec.ExitError`
- Log: `[session] process exited for thread %d (session %s): exit code %d, signal %s`
- Store exit info in Session struct for the error message in SendMessage

### 3. Common Crash Causes

Investigate and handle these known causes:

**OOM Kill:** On Raspberry Pi with limited RAM, Claude Code can be OOM-killed. Check if the exit signal is SIGKILL (9). If so, include "possible OOM kill" in the error message. Consider logging memory usage of claude processes.

**Broken pipe / stdin write failure:** If the process dies between messages, the next `stdin.Write()` at pool.go:290 will fail. This path already calls `markDead()` — but the error message should mention the underlying cause.

**Scanner buffer overflow:** The scanner has a 1MB buffer (pool.go:213). If Claude outputs a single line >1MB, the scanner fails silently. Check if `s.scanner.Err()` catches this and surface the error.

**Session file corruption:** If the Claude session .jsonl file is corrupted, Claude Code may crash on `--resume`. The "No conversation found" error is already handled, but other session file errors may not be.

### 4. Automatic Recovery

Currently when the process dies, the user sees the error and must send a new message (which spawns a new session). Improve this:

**Fix:**
- When `SendMessage` detects a dead process, instead of just returning an error, attempt ONE automatic retry: spawn a new session and resend the message
- Add a `retried` flag to prevent infinite retry loops
- Log the retry attempt: `[session] auto-retrying message after process death for thread %d`
- If the retry also fails, return the error to the user with full diagnostics
- Clear the session ID from the thread if the process keeps crashing (so next attempt starts fresh instead of resuming a potentially corrupt session)

### 5. Proactive Health Monitoring

Add lightweight monitoring to detect dying processes earlier:

- After the process exit goroutine (pool.go:227) detects death, if a message is in-flight (s.busy == true), immediately unblock the scanner by closing stdout
- This prevents SendMessage from hanging indefinitely on a dead process's scanner

- Test automatic retry: simulate death, verify retry spawns new session and succeeds

- Do NOT change the subprocess spawning args or the NDJSON protocol
- Focus on observability (better errors) and resilience (auto-retry)
- All fixes must pass `make check`
