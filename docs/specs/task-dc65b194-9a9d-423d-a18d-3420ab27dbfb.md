The live task output view shows "RECONNECTING" indefinitely instead of streaming the actual task progress. The SSE connection for task output keeps dropping and reconnecting without ever displaying content.

## Current behavior
- User opens a running task's detail page to watch live output
- Instead of seeing Claude Code's streaming output, the UI shows "RECONNECTING" status
- The connection never stabilizes — it keeps cycling through reconnect attempts

## Expected behavior
- When viewing a running task, the live output streams in real-time
- The SSE connection stays open and stable for the duration of the task
- Output lines appear as Claude Code produces them (tool calls, text, etc.)

## Investigation areas

### Backend — SSE endpoint for task output
- Check the task output SSE endpoint (likely `GET /api/v1/tasks/:id/output/stream` or similar)
- Verify the endpoint correctly:
  - Sets `Content-Type: text/event-stream` header
  - Sets `Cache-Control: no-cache` and `Connection: keep-alive`
  - Flushes data after each event (uses `c.Writer.Flush()` in Gin)
  - Keeps the connection alive with periodic heartbeat/keepalive events (e.g. SSE comment `: keepalive\n\n` every 15-30 seconds)
- Check if the endpoint returns immediately (404, error) instead of staying open — this would cause the frontend to reconnect in a loop

### Backend — Task output capture
- Verify the task executor (`internal/runner/executor.go`) is actually capturing and publishing output
- Check if there's a pub/sub mechanism or if the SSE handler reads from a buffer/channel
- Make sure output is being written to wherever the SSE endpoint reads from

### Frontend — SSE reconnect logic
- Check the EventSource or SSE client implementation for task output
- If using a reconnect loop, ensure it has:
  - Backoff between retries (not hammering the server)
  - Detection of "connected but no data" vs "connection refused"
  - Proper cleanup on component unmount
- The "RECONNECTING" state might be shown prematurely — verify the condition that triggers this display

### Possible root causes
1. **SSE endpoint returns immediately** — handler doesn't block/stream, so frontend sees connection close and reconnects
2. **Missing flush** — data is buffered and never sent to the client
3. **CORS or proxy issue** — if Vite dev server proxies `/api`, SSE streams might be buffered
4. **No keepalive** — connection times out due to inactivity between output lines
5. **Output not being captured** — executor runs but output goes nowhere

## Implementation Notes
- Compare with the working chat SSE stream implementation — chat streaming works, so use that as a reference
- The chat SSE uses `c.Stream()` or `c.SSEvent()` in Gin — task output should use the same pattern
- Check Vite proxy config (`frontend/vite.config.ts`) — SSE needs special handling (no response buffering)
- Existing tests must pass (`make check`)
