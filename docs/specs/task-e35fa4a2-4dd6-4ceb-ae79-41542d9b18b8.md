# Fix AskUserQuestion Panel Not Displaying

The AskUserQuestion tool call panel never appears in the chat UI. The component (`frontend/src/components/AskUserPanel.tsx`) exists and is wired into ChatView, but it never actually renders for users. This needs investigation and fixing.

## Architecture Overview

The AskUserQuestion flow spans several layers:

1. **Claude subprocess** (pool.go) outputs a `tool_use` NDJSON event with name "AskUserQuestion"
2. **Backend handler** (chat.go:617-629) parses it and sends `event: tool_use` SSE event to frontend
3. **Frontend SSE parser** (client.ts:595) yields `{ tool_use: { id, name, input } }` chunk
4. **SSE Context** (SSEContext.tsx:327-334) adds to `session.state.toolCalls` array
5. **ChatView** (ChatView.tsx:811-831) renders AskUserPanel when `isStreamingThisThread && activeToolCalls.length > 0 && tc.name === 'AskUserQuestion'`
6. **AskUserPanel** submits the answer via POST `/threads/:id/tool-results`
7. **Backend** (chat.go:1099) calls `claude.Sessions.SendToolResult()` which writes NDJSON to Claude's stdin

## Likely Root Causes (investigate all)

### 1. SSE Connection Drops During Tool Wait (Most Likely)

When Claude calls AskUserQuestion, the process waits for a tool_result on stdin. During this wait:
- The scanner in `pool.go:300` is blocked on `s.scanner.Scan()` — no output
- The backend SSE stream sends no data
- The HTTP connection is idle
- **Nginx/reverse proxy or browser may timeout the idle SSE connection**

If the connection drops:
- Frontend loses the stream
- `retryStreamFn` reconnects to `/threads/:id/stream` (the StreamBuffer replay endpoint)
- But the StreamBuffer likely doesn't have the pending tool_use event (it was already sent)
- After reconnect, `isStreaming` may become false or `toolCalls` is empty
- AskUserPanel never renders

**Fix:** Send periodic SSE keepalive comments during tool waits. In `chat.go`, inside the `for event := range stream` loop, add a timeout mechanism: if no event arrives for 15 seconds, send a `:keepalive\n\n` SSE comment (SSE spec says lines starting with `:` are comments, ignored by EventSource but keep the connection alive).

Implementation approach:
```go
// In streamEventsToClient, replace the simple range loop with a select
for {
    select {
    case event, ok := <-stream:
        if !ok {
            goto done
        }
        // ... existing event handling ...
    case <-time.After(15 * time.Second):
        // Send SSE keepalive to prevent proxy/browser timeout
        if !clientDisconnected {
            fmt.Fprint(c.Writer, ":keepalive\n\n")
            flusher.Flush()
        }
    case <-clientGone:
        clientDisconnected = true
    }
}
```

### 2. AskUserPanel Only Renders During Streaming

In ChatView.tsx:811, the panel only renders when `isStreamingThisThread` is true:
```tsx
{isStreamingThisThread && (streamingContent || streamingThinking || activeToolCalls.length > 0) && (
```

If the stream completes or errors for any reason, AskUserPanel immediately disappears — even if the tool hasn't been answered yet.

**Fix:** Track pending tool calls separately from streaming state. A tool call with no `result` is "pending". Render pending AskUserQuestion panels regardless of streaming state:

```tsx
// Always render unanswered AskUserQuestion panels, even after stream ends
{pendingAskUserCalls.length > 0 && (
    <div className="mb-2 max-w-3xl">
        {pendingAskUserCalls.map(tc => (
            <AskUserPanel key={tc.id} toolCall={tc} threadId={threadId} />
        ))}
    </div>
)}
```

Where `pendingAskUserCalls` filters `activeToolCalls` for `name === 'AskUserQuestion' && !result`.

### 3. Reconnection Clears Tool State

In SSEContext.tsx:222-225, when reconnecting after connection loss, `content` and `thinking` are cleared but `toolCalls` is NOT cleared. However, the reconnection stream won't replay tool_use events — the tool_use event was already consumed.

After reconnection:
- `toolCalls` still has entries from before the disconnect (good)
- But `isStreaming` may flip to false during reconnect attempts (bad — AskUserPanel vanishes)
- If reconnection ultimately fails, `isStreaming` becomes false permanently

**Fix:** Persist pending tool calls across reconnection. The fix from #2 (rendering outside streaming guard) solves this.

### 4. Tool Result Submission May Fail Silently

In AskUserPanel.tsx:97-108, `handleSubmit` calls `submitToolResult()` but only logs errors to console. If the submission fails (e.g., session died), the user gets no feedback and Claude never receives the answer.

**Fix:**
- Show a user-visible error toast/message if `submitToolResult` fails
- Add a retry button for failed submissions
- If the error indicates the session is dead, show a message like "Session expired. Please resend your message."

### 5. Tool Call Persistence Across Page Navigation

If the user navigates away from the thread and back while AskUserQuestion is pending, the SSE session state (including toolCalls) is lost. There's no way to recover the pending tool call.

**Fix:** When saving tool_use events to `collectedToolCalls` in the chat handler (chat.go:632), also check if the tool is AskUserQuestion. If so, store the pending state in the database (or a dedicated field on the thread). When the chat page loads, check for pending AskUserQuestion calls and display them.

Alternatively (simpler): store pending tool calls in the SSE session state that persists via the StreamBuffer. When a client reconnects, replay any unanswered tool_use events.

## Testing

After fixing, verify:
1. Send a message that triggers AskUserQuestion (use a Claude prompt that asks for clarification)
2. The AskUserPanel appears immediately with options and/or free text input
3. Wait 30+ seconds — the panel stays visible (keepalive prevents timeout)
4. Select an option and submit — Claude receives the answer and continues
5. Refresh the page while AskUserQuestion is pending — panel should reappear (stretch goal)
6. Navigate away and back — panel should reappear (stretch goal)

## Priority Order

Fix in this order for maximum impact:
1. SSE keepalive (#1) — prevents the most common failure
2. Render outside streaming guard (#2) — makes panel resilient to state issues
3. Error feedback on submit (#4) — user knows when something fails
4. Persistence (#5) — nice to have but complex

## Important

- The existing AskUserPanel component UI looks good — don't redesign it
- Don't change the Claude subprocess protocol (NDJSON format)
- The `submitToolResult` API endpoint and `SendToolResult` backend method work correctly — the issue is upstream (panel not displaying, not submission)
- All fixes must pass `make check`
