# Persistent Claude Code sessions via stream-json protocol

Replace the per-message process spawn model with persistent multi-turn sessions using Claude Code's `--input-format stream-json` protocol. A single process stays alive across all messages in a thread, eliminating process restarts and token re-loading between messages.

## Background

Currently, each chat message spawns a new Claude Code process (or reuses a pre-warmed one from the pool). Even with `--resume`, each process reloads the full session context from disk, costing tokens. The `--input-format stream-json` protocol allows sending multiple user messages to a single long-lived process via NDJSON on stdin.

## Requirements

### Core: Persistent session manager
- Replace the session pool (`internal/claude/pool.go`) with a session manager that tracks live, persistent Claude Code processes per thread
- Launch process with: `claude --output-format stream-json --input-format stream-json --verbose --dangerously-skip-permissions [--resume <id>] [--session-id <id>] [--model <model>] [--append-system-prompt-file <path>]`
- Do NOT use `-p` flag — the process enters long-lived mode reading NDJSON from stdin
- Process stays alive between messages, waiting for next NDJSON input on stdin
- Idle timeout: kill process after 5 minutes of no messages (configurable, same as current pool TTL)
- Reset idle timer on each message

### Input protocol (stdin to Claude Code)
- User messages: `{"type":"user","message":{"role":"user","content":"<prompt>"}}\n`
- Keep-alive: `{"type":"keep_alive"}\n` (optional, for connection health)
- End session: `{"type":"control_request","request_id":"<uuid>","request":{"subtype":"end_session"}}\n`

### Output protocol (Claude Code to stdout)
- Parse NDJSON lines from stdout, same event types as current (`system`, `assistant`, `stream_event`, `result`)
- A `{"type":"result",...}` line marks end of a turn — process is then idle and ready for next message
- The existing `parseEvent()` function in `runner.go` should work with minimal changes since the output format is the same `stream-json`

### Chat handler changes (`internal/handlers/chat.go`)
- In `streamResponse`: instead of spawning a new process or acquiring from pool, get-or-create a persistent session from the session manager
- If no session exists for the thread, start a new process (with context assembly for new sessions, `--resume` for existing ones)
- Write the user message as NDJSON to the process's stdin
- Read events from stdout until `result` event (end of turn)
- Do NOT close stdin after writing — the process stays alive
- Do NOT unregister from Registry after response — session persists
- Multiple concurrent messages to same thread: queue them (only one message at a time per process)

### Session lifecycle
- **Start:** First message to a thread creates the process
- **Message:** Write NDJSON user message, read until result event
- **Idle:** Process stays alive, idle timer ticking
- **Timeout:** After 5 min idle, kill process and unregister from Registry
- **Manual kill:** Registry.Kill sends `end_session` control request, then kills process
- **Error/crash:** Detect process exit, clean up session manager entry, next message starts a fresh process
- **Thread delete:** Evict and kill session
- **Session clear:** Kill process, next message starts fresh with new session ID

### Session invalidation (must restart process)
- Model change on thread: kill old process, next message starts new one with new model
- Project change on thread: kill old process (different working directory)
- Session clear / new session: kill old process

### Registry integration
- Register process once at creation, unregister only on kill/crash/timeout
- `startedAt` stays stable across messages — no more duration resets
- `/api/v1/processes` shows truly persistent sessions

### Concurrency safety
- Session manager must be thread-safe (mutex around session map)
- Only one message can be in-flight per session at a time — use a per-session mutex or channel
- If a second message arrives while one is streaming, queue it or return HTTP 409

## Implementation Notes

- The session manager replaces `SessionPool` in `pool.go` — rewrite the file rather than patching
- Keep `Run` function working as fallback for edge cases (e.g. compaction)
- The `--input-format stream-json` flag requires `--verbose`; do NOT combine with `-p`
- First message in a session should still use `--append-system-prompt-file` for context assembly
- Subsequent messages need only the user content — Claude maintains conversation state in memory
- `--resume` works with stream-json to resume an existing session from disk
- Test with: `echo '{"type":"user","message":{"role":"user","content":"hello"}}' | claude --output-format stream-json --input-format stream-json --verbose --dangerously-skip-permissions`
- Existing tests must pass after changes (`make check`)
- Reference the Python SDK implementation at `/home/pi/.local/lib/python3.12/site-packages/claude_code_sdk/` for protocol details

## Out of scope
- Control requests beyond `end_session` (e.g. `set_model`, `interrupt`) — can be added later
- Task runner changes — task mode stays as-is (batch execution, different lifecycle)

## Safety

**NEVER run `make deploy`, `make install-service`, `systemctl restart botka`, or any command that would restart the Botka service.** Task agents run inside Botka — deploying would kill the agent's own process.
