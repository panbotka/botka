When a chat is unpinned, it stays at the top of the unpinned list instead of sorting by last message time. Fix the sort order so threads are ordered by the datetime of their last message.

## Root Cause

In `internal/handlers/thread.go:74`, the thread list query sorts by:

```go
query = query.Order("t.pinned DESC, t.updated_at DESC")
```

The problem is `t.updated_at` — when a thread is unpinned, the `updated_at` column gets bumped (GORM auto-updates it on any row change), so the unpinned thread appears at the top.

The query already computes `last_message_at` via a lateral join (line 64-69):

```go
Select(`t.*, lm.content AS last_message_preview, lm.created_at AS last_message_at`).
Joins(`LEFT JOIN LATERAL (
    SELECT LEFT(content, 150) AS content, created_at
    FROM messages WHERE thread_id = t.id
    ORDER BY created_at DESC LIMIT 1
) lm ON true`)
```

## Fix

Change the ORDER BY on line 74 from:

```go
query = query.Order("t.pinned DESC, t.updated_at DESC")
```

to:

```go
query = query.Order("t.pinned DESC, COALESCE(lm.created_at, t.created_at) DESC")
```

This sorts by:
1. Pinned threads first
2. Then by last message time (falling back to thread creation time if no messages)

Using `COALESCE` handles threads with no messages yet (newly created).

## Also check the frontend

The frontend in `frontend/src/utils/threadOrder.ts` (`getSidebarThreads`) currently just separates pinned from regular without re-sorting. Since the backend returns them in the correct order, this should be fine — but verify the frontend doesn't re-sort or shuffle the order anywhere.

Also check `frontend/src/components/ThreadSidebar.tsx` — the `pinnedThreads`, `regularThreads`, and `archivedThreads` are filtered from the threads array (lines 235-237). Verify they preserve the backend sort order (`.filter()` preserves order, so this should be OK).
