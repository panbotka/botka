## Problem

When navigating to `/tasks` from the menu, the page always defaults to "All" tab. The user wants to see the most relevant/actionable tasks first without clicking through tabs.

## Requirements

When opening the Tasks page without an explicit `?status=` URL parameter, auto-select the first tab that has tasks, in this priority order:

1. **Running** — if any tasks are running, show those
2. **Queued** — if nothing running, show queued
3. **Pending** — if nothing queued, show pending
4. **All** — fallback if none of the above have tasks

This should only apply on initial page load without URL parameters. If the user navigates with `?status=done` or clicks a tab manually, respect that choice.

## Implementation

In `frontend/src/pages/TasksPage.tsx`:

1. On initial load (no `?status=` param), fetch task counts per status (or use the existing task list data)
2. Determine the first non-empty status from the priority list: running → queued → pending → all
3. Set that as the active filter
4. If the user clicks a tab manually or arrives with a URL param, use that instead

The logic should be efficient — avoid multiple API calls if possible. If the existing task list fetch already returns all statuses, just count from that data.

## Testing

- Navigate to `/tasks` with no params — should show running tab if any running, else queued, else pending, else all
- Navigate to `/tasks?status=done` — should show done tab regardless
- Click a tab manually — should stay on that tab
- Run `npx tsc --noEmit` and `npx eslint src/` from the frontend directory

## Safety

**NEVER run `make deploy`, `make install-service`, `systemctl restart botka`, or any command that would restart the Botka service.** Task agents run inside Botka — deploying or restarting would kill the agent's own process. Only build and test.
