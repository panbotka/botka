## Problem

`ChatInput` component has local state (`useState('')` for input text, `useState<File[]>([])` for files). When switching threads, React reuses the same component instance — the input text and attached files persist across thread switches instead of being independent per thread.

## Root Cause

In `ChatView.tsx` (~line 1017), `ChatInput` is rendered without a `key` prop tied to the thread. React reconciliation reuses the instance, so its internal state survives thread changes.

## Fix

Add `key={threadId}` to the `ChatInput` component in `ChatView.tsx` so React unmounts and remounts it when the thread changes. This gives each thread a fresh input.

### File: `frontend/src/components/ChatView.tsx`

Find where `<ChatInput` is rendered (~line 1017) and add the thread-based key:

```tsx
<ChatInput
  key={threadId}    // <-- add this
  ref={chatInputRef}
  onSend={handleSend}
  ...
/>
```

The `threadId` prop is already available in ChatView (it's a component prop).

## Notes

- This is the simplest and most React-idiomatic solution — no need to lift state or add controlled input.
- Side effect: switching away from a thread loses the draft text. This is acceptable — the user expects independent inputs.
- The `ref={chatInputRef}` will also be recreated, which is fine since ChatView already handles ref updates.
- Run `make check` to verify.
