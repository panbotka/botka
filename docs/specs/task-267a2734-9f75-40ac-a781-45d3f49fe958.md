Fix font and background colors that break in dark mode due to insufficient contrast.

## Context

Dark mode works by swapping the `zinc` color scale in `frontend/src/index.css:19-30` — in dark mode, `zinc-50` becomes `#18181b` (dark) and `zinc-900` becomes `#fafafa` (light). This means all `text-zinc-*` and `bg-zinc-*` classes automatically flip correctly.

**However**, the following color types are NOT remapped and cause contrast issues in dark mode:

### 1. Hardcoded `bg-white` (81 occurrences across 21 files)
`bg-white` stays white in dark mode, creating bright boxes on a dark background. These need to be changed to `bg-zinc-50` (which resolves to dark in dark mode) or similar zinc equivalents.

Key files with `bg-white`:
- `ThreadSidebar.tsx` (6) — dropdown menus, focus states
- `SettingsPage.tsx` (22) — cards, inputs, modals
- `ChatPage.tsx` (6) — main layout
- `ProjectsPage.tsx` (5) — cards
- `DashboardPage.tsx` (4) — cards
- `MessageBubble.tsx` (5) — message backgrounds
- `TaskDetailPage.tsx` (5) — cards
- `Lightbox.tsx` (6) — overlay
- `ModelPicker.tsx` (3) — dropdown
- `CommandPalette.tsx` (2) — palette background
- `ChatInput.tsx` (3) — input area
- `TaskList.tsx` (3) — drag items
- `RunnerStatus.tsx` (2) — status cards

### 2. Hardcoded `text-white` used on colored backgrounds
Some `text-white` is used on colored badges/buttons (e.g., `bg-emerald-600 text-white`). These are probably fine since the background provides contrast. But `text-white` used on zinc backgrounds will become invisible in dark mode. Audit each usage.

### 3. Non-zinc accent colors (emerald, amber, red, etc.)
Colors like `text-emerald-600`, `bg-emerald-50`, `text-red-600`, `bg-amber-100`, `text-amber-500` are not remapped. In dark mode:
- Light backgrounds like `bg-emerald-50`, `bg-amber-100`, `bg-red-50` will stay light — creating bright patches
- Dark text colors like `text-emerald-600` on a now-dark background might be OK or might need lightening

### 4. Inline styles and hex colors
Search for `style={{` with color/background values and hardcoded hex colors. Tag colors in ThreadSidebar use inline `backgroundColor` — these won't flip.

### 5. CSS file (`index.css`)
- Line 64: `text-emerald-600` for links — may need to be lighter in dark mode
- Line 68: `text-zinc-500` for blockquotes — this should flip via the variable swap (OK)
- Line 80: `bg-zinc-100` for table headers — should flip (OK)
- `[&_mark]:bg-amber-100 [&_mark]:text-amber-800` in search results — won't flip

## Approach

**Do NOT add Tailwind `dark:` variants** — the app uses CSS variable swapping, not Tailwind's dark mode class strategy. Instead:

1. **Replace `bg-white`** with `bg-zinc-50` everywhere (or `bg-zinc-100` for elevated surfaces). Since `zinc-50` maps to `#18181b` in dark mode, this automatically works.

2. **Replace `focus:bg-white`** with `focus:bg-zinc-50`.

3. **For non-zinc accent colors**, add CSS variable overrides in `:root.dark` in `index.css`, similar to how zinc is handled. At minimum, add overrides for the most-used accent scales:
   - `emerald-50` through `emerald-600` (used extensively for accents, links, badges)
   - `amber-50` through `amber-600` (used for pins, warnings, highlights)
   - `red-50` through `red-600` (used for delete confirmations, errors)
   
   For each, swap light↔dark shades so e.g. `bg-emerald-50` becomes dark in dark mode and `text-emerald-600` becomes lighter.

4. **For `shadow-lg shadow-zinc-200/50`** — in dark mode this should be darker. Consider overriding with `shadow-zinc-900/50` or similar by adding the shadow colors to the CSS variable swap.

5. **For inline `style` colors** (tag colors, etc.) — these are user-defined and can't be swapped. They're acceptable as-is since they're small decorative elements.
