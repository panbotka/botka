# Fix Command Execution for Dev Scripts That Spawn Background Processes

## Problem

The current command execution (`POST /api/v1/projects/:id/run-command`) runs `bash -c "<command>"` and tracks the bash shell PID. This breaks for dev scripts that build and then launch a server via `nohup ... &` — the bash process exits immediately after spawning the background server, so:

1. Botka tracks the dead shell PID
2. `isProcessAlive` returns false → UI shows no running command
3. The actual server process runs on a different PID that Botka doesn't know about

Example: RUNWATCH's `make dev` runs `dev.sh` which does `nohup ./runwatch serve ... &` — the script exits, but the server keeps running.

## Solution

Instead of tracking the bash PID, detect what's actually running after the command finishes. The most reliable approach:

### Add a `port` field to project commands

Each command can optionally specify a port. If a port is configured, Botka can:
- After the command's bash process exits, check if something is listening on that port
- Track the PID of whatever process is on that port (via `lsof -ti :<port>`)
- For the "alive" check, verify the port is still in use rather than just checking the original PID

### Model Changes

Add two new nullable integer columns to the `projects` table via migration:
- `dev_port` — port number the dev server listens on (e.g. 8093)
- `deploy_port` — port number for deploy (usually same as dev, or null for deploy-and-exit commands)

Add to the `Project` model:
- `DevPort *int` with `gorm:"type:integer" json:"dev_port"`
- `DeployPort *int` with `gorm:"type:integer" json:"deploy_port"`

### Backend Changes (`internal/handlers/command.go`)

Modify `RunCommand`:
1. Start the bash command as before
2. Start a goroutine that waits for the command to complete (`cmd.Wait()`)
3. After the command exits, if a port is configured for this command type:
   - Wait up to 5 seconds, checking every 500ms if something is listening on the port
   - If found, get the PID via `lsof -ti :<port>` and update the tracked `RunningCommand` with the real PID
   - If not found, mark the command as completed/failed
4. If no port is configured, behave as today (fire-and-forget, track bash PID)

Modify `RunningCommand` struct to include an optional `Port` field:
```go
type RunningCommand struct {
    PID         int       `json:"pid"`
    Port        int       `json:"port,omitempty"`
    CommandType string    `json:"command_type"`
    Command     string    `json:"command"`
    ProjectID   uuid.UUID `json:"project_id"`
    StartedAt   time.Time `json:"started_at"`
}
```

Modify `isProcessAlive` / `ListCommands`:
- If the command has a port, check aliveness by verifying the port is still in use (try `net.DialTimeout("tcp", fmt.Sprintf(":%d", port), time.Second)` or use `lsof`)
- Fall back to PID signal-0 check if no port is set

Modify `KillCommand`:
- If the command has a port, also look up and kill the process on that port (to handle PID changes after restart)
- Kill the process group as before

### Frontend Changes

#### Settings UI (ProjectDetailPage.tsx, ProjectsPage.tsx)
- Add port number inputs next to the dev command and deploy command fields
- Label: "Port" with a small numeric input (placeholder: "8093")
- Save alongside the command strings via the existing project update API

#### Chat header buttons and ProjectDetailPage CommandsSection
- Show the port number in the running status if available: "Dev running on :8093 (PID 12345)"
- No other changes needed — the buttons already use the same API

#### Types
- Add `dev_port?: number` and `deploy_port?: number` to the Project type

### Project Update API
- Accept `dev_port` and `deploy_port` in the update request (nullable integers)
- Validate: if provided, must be between 1 and 65535

## Edge Cases

- If the port is already in use BEFORE running the command (e.g. previous dev instance), the dev script should handle killing it (like RUNWATCH's dev.sh already does). Botka just needs to track whatever ends up on the port after the command runs.
- If multiple processes listen on the same port (shouldn't happen), pick the first one from lsof.
- Deploy commands typically don't need a port (they build and restart a systemd service). A null port means "track the bash process PID only, old behavior".
- If the command fails (bash exits non-zero) and nothing is on the port, remove the tracked command and report failure.
