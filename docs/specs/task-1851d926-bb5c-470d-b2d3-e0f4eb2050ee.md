## Problem

Searching messages in chat crashes with a JavaScript error:

```
Uncaught TypeError: can't access property "title", A.thread is undefined
```

The search results reference a `thread` object that is undefined — likely the API returns search results with a `thread` relation that isn't being populated/included, or the frontend assumes `thread` is always present when it may be null.

## Investigation

1. Find the search endpoint in the backend — likely in `internal/handlers/` (messages or search handler)
2. Check if the search response includes the `thread` relation (GORM preload/joins)
3. Find the frontend component that renders search results — look for `.thread.title` access
4. Determine if the fix should be backend (include thread in response) or frontend (null-safe access) or both

## Likely Fix

**Backend:** The search endpoint probably doesn't preload the Thread association. Add `.Preload("Thread")` to the GORM query.

**Frontend:** Add null-safe access as a defensive measure: `result.thread?.title` instead of `result.thread.title`.

## Testing

- Open chat UI
- Use the search function to search for messages
- Verify results display correctly with thread titles
- Verify searching for terms that match messages in deleted threads doesn't crash
- Run `make check`
- Run `npx tsc --noEmit` from the frontend directory

## Safety

**NEVER run `make deploy`, `make install-service`, `systemctl restart botka`, or any command that would restart the Botka service.** Task agents run inside Botka — deploying or restarting would kill the agent's own process. Only build and test.
