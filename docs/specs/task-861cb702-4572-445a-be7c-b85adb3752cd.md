Add authentication to Botka. There is only ever one user — no registration flow needed. Support password login and WebAuthn passkeys.

## Requirements

### Database — User and session models
- Create a `users` table (migration): `id` (bigserial), `username` (unique), `password_hash` (bcrypt), `created_at`, `updated_at`
- Create a `sessions` table: `id` (text, random token), `user_id`, `expires_at`, `created_at`
- Create a `webauthn_credentials` table: `id` (bigserial), `user_id`, `credential_id` (bytea, unique), `public_key` (bytea), `sign_count` (int), `name` (text, e.g. "iPhone passkey"), `created_at`
- Seed the initial user via migration or a startup check — if no user exists, create one with:
  - Username: `botka`
  - Password: generate a random secure password (16+ chars, alphanumeric + special)
  - Print the generated credentials to stdout on first run: `[auth] Created initial user: botka / <password>`
  - Also write them to a file `/tmp/botka-initial-credentials.txt` so the operator can retrieve them

### Backend — Password auth endpoints
- `POST /api/v1/auth/login` — accepts `{"username": "...", "password": "..."}`, returns session token as HTTP-only cookie
- `POST /api/v1/auth/logout` — clears session cookie
- `GET /api/v1/auth/me` — returns current user info if authenticated, 401 if not
- `PUT /api/v1/auth/password` — change password (requires current password)
- Session tokens: random 64-char hex string, stored in `sessions` table with expiry (30 days)
- Cookie: `botka_session`, HTTP-only, SameSite=Lax, Secure (if HTTPS), Path=/

### Backend — WebAuthn passkey endpoints
- Use the `go-webauthn/webauthn` library (github.com/go-webauthn/webauthn)
- `POST /api/v1/auth/passkey/register/begin` — start passkey registration (returns challenge)
- `POST /api/v1/auth/passkey/register/finish` — complete registration (stores credential)
- `POST /api/v1/auth/passkey/login/begin` — start passkey login (returns challenge)
- `POST /api/v1/auth/passkey/login/finish` — complete login (verifies signature, creates session)
- `GET /api/v1/auth/passkeys` — list registered passkeys (id, name, created_at)
- `DELETE /api/v1/auth/passkeys/:id` — remove a passkey
- WebAuthn challenges should be stored temporarily (in-memory map with TTL, or in a `webauthn_challenges` table)
- The User model must implement `webauthn.User` interface (ID, Name, DisplayName, Credentials)

### Backend — Auth middleware
- Create a Gin middleware that checks the `botka_session` cookie against the sessions table
- Protected routes: ALL `/api/v1/*` endpoints EXCEPT `/api/v1/auth/login`, `/api/v1/auth/passkey/login/*`, and `/api/v1/auth/me`
- Also exempt: `/mcp/*` (MCP uses its own auth), static files, health check
- On 401: return `{"error": "unauthorized"}`
- Middleware sets the user in Gin context (`c.Set("user", user)`)

### Frontend — Login page
- Create a `/login` page with:
  - Username + password form
  - "Sign in with passkey" button (if WebAuthn is supported in the browser)
  - Clean, centered card layout, matching app design language
- After successful login, redirect to the page the user was trying to access (or `/` by default)
- If WebAuthn is available and passkeys are registered, show the passkey option prominently

### Frontend — Auth context and route protection
- Create an `AuthContext` that:
  - Calls `GET /api/v1/auth/me` on mount to check if logged in
  - Provides `isAuthenticated`, `user`, `login()`, `logout()` to children
  - Shows a loading state while checking auth
- Wrap all app routes in a `ProtectedRoute` component that redirects to `/login` if not authenticated
- The login page is the only public route

### Frontend — Passkey management in Settings
- Add a "Security" section to the Settings page:
  - "Change password" form
  - "Passkeys" section:
    - List registered passkeys (name, created date)
    - "Add passkey" button that triggers WebAuthn registration flow
    - Delete button for each passkey
  - WebAuthn registration uses `navigator.credentials.create()` and `navigator.credentials.get()`

### Frontend — Auto-redirect on 401
- If any API call returns 401, redirect to the login page
- Clear the auth context state

## Environment Variables
- `WEBAUTHN_ORIGIN` — the app's origin for WebAuthn (e.g. `https://botka.example.com`), defaults to `http://localhost:5110`
- `WEBAUTHN_RPID` — Relying Party ID (e.g. `botka.example.com`), defaults to `localhost`
- `SESSION_MAX_AGE` — session duration, defaults to `720h` (30 days)

## Implementation Notes
- Use `golang.org/x/crypto/bcrypt` for password hashing (already commonly used in Go)
- Use `github.com/go-webauthn/webauthn` v2 for WebAuthn — it handles all the crypto
- The session cookie approach is simple and works well with both SSR and SPA
- For the frontend WebAuthn API, use `@simplewebauthn/browser` npm package for easier integration, or raw `navigator.credentials` API
- Store initial credentials also in stdout so the operator sees them on first deploy
- Existing tests must pass (`make check`)
- Write tests for the auth middleware and login endpoint

## Credential storage
After the task completes, the operator will manually store the credentials in 1Password. The task should just make them available via stdout + temp file.

## Safety
**NEVER run `make deploy`, `make install-service`, `systemctl restart botka`, or any command that would restart the Botka service.**
