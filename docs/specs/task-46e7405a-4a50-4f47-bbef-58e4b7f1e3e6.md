# Global search (Cmd+K)

Implement a global search overlay accessible via Cmd+K keyboard shortcut, similar to Slack's quick switcher. Searches across tasks, projects, threads, and chat messages with results grouped by type.

## Requirements

### Backend — Search endpoint
- Add `GET /api/v1/search?q=<query>&limit=<N>` endpoint
- Search across 4 entity types in priority order: Tasks, Projects, Threads (by title), Messages (by content)
- Use PostgreSQL `unaccent()` extension + case-insensitive `ILIKE` for matching (handles diacritics transparently)
- Return results grouped by type, each group with max 5 items by default
- Response shape: `{"data": {"tasks": [...], "projects": [...], "threads": [...], "messages": [...]}}`
- Each result item includes: `id`, `title` (or content snippet for messages), `type`, and enough info to navigate (e.g. `thread_id` for messages, `project_name` for tasks)
- Minimum query length: 2 characters
- Ensure `unaccent` PostgreSQL extension is created via migration if not already present

### Backend — Search logic
- Tasks: search in `title` and `spec` fields, exclude deleted tasks
- Projects: search in `name` field
- Threads: search in `title` field
- Messages: search in `content` field, return parent thread_id and a short snippet (~100 chars) around the match
- All searches use `unaccent(lower(column)) LIKE unaccent(lower(%query%))` pattern
- Order results within each group by relevance: prefer title matches over body matches, then by recency (`updated_at DESC`)

### Frontend — Keyboard shortcut
- Register global `Cmd+K` (Mac) / `Ctrl+K` (other) shortcut that opens the search overlay
- `Escape` closes it, clicking outside closes it
- If already open, `Cmd+K` focuses the input

### Frontend — Search overlay UI
- Modal overlay with a large search input at the top (auto-focused)
- Results appear below the input, grouped under section headers: "Tasks", "Projects", "Threads", "Messages"
- Empty sections are hidden
- Each result row shows: icon (per type), title/name, and subtle metadata (e.g. project name for tasks, thread title for messages)
- Keyboard navigation: arrow keys to move between results, Enter to navigate
- Debounce input by ~300ms before firing API request
- Show a subtle loading indicator while searching
- Clicking a result navigates to: task detail, project page, thread, or thread (scrolled to message)

### Frontend — Styling
- Follow existing app design language (Tailwind, zinc color palette, consistent with current UI)
- Overlay should have a semi-transparent backdrop
- Search input should be prominent, results compact

## Implementation Notes

- Create migration to enable `unaccent` extension: `CREATE EXTENSION IF NOT EXISTS unaccent`
- For the search handler, create a new file `internal/handlers/search.go`
- For the frontend overlay, create a new component `frontend/src/components/SearchOverlay.tsx`
- Debounce with a simple `setTimeout`/`clearTimeout` in a custom hook or inline — no need for a library
- Existing tests must pass after changes (`make check`)

## Safety

**NEVER run `make deploy`, `make install-service`, `systemctl restart botka`, or any command that would restart the Botka service.** Task agents run inside Botka — deploying would kill the agent's own process.
