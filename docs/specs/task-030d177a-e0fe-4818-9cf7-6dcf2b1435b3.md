Add a dedicated list of URL sources per chat thread. When a session is created, all URLs are fetched and their content is included in the context sent to Claude, alongside the existing system prompt.

## Requirements

### Database — Thread sources
- Create a `thread_sources` table (migration):
  - `id` (bigserial)
  - `thread_id` (bigint, FK to threads)
  - `url` (text, not null)
  - `label` (text, optional — user-friendly name like "React docs")
  - `position` (int — ordering)
  - `created_at`, `updated_at`
- A thread can have 0..N sources
- Add GORM model `ThreadSource` with belongs-to Thread relation

### Backend — CRUD endpoints
- `GET /api/v1/threads/:id/sources` — list sources for a thread (ordered by position)
- `POST /api/v1/threads/:id/sources` — add a source `{"url": "...", "label": "..."}`
- `PUT /api/v1/threads/:id/sources/:source_id` — update url/label
- `DELETE /api/v1/threads/:id/sources/:source_id` — remove a source
- `PUT /api/v1/threads/:id/sources/reorder` — reorder sources `{"ids": [3, 1, 2]}`

### Backend — Fetch sources during context assembly
- In `internal/claude/context.go` (or wherever the session context is assembled), after the existing context steps:
  1. Load thread sources from DB
  2. Fetch each URL (HTTP GET, timeout 15s per URL, total timeout 60s)
  3. Extract text content from HTML (strip tags, get main content — use a simple approach like extracting `<body>` text or `<main>`/`<article>` if present)
  4. For non-HTML content (plain text, JSON, markdown): include as-is
  5. Include fetched content in the context with clear delimiters:
     ```
     ## Source: React Documentation (https://react.dev/reference)
     <fetched content, truncated to max 50k chars per source>
     ```
  6. If a fetch fails, include a note: `## Source: <label> (<url>) — FETCH FAILED: <error>`
  7. Total source content budget: max 200k chars across all sources (truncate oldest/longest if exceeded)
- Sources are fetched fresh on every session creation (they might change)
- Cache fetched content for 5 minutes to avoid re-fetching on rapid session restarts

### Frontend — Sources UI in thread settings
- Add a "Sources" section in the thread settings/config area (wherever system prompt is edited)
- Show it alongside (not replacing) the system prompt editor
- List of sources with:
  - URL (editable text input)
  - Label (optional, editable)
  - Delete button (X icon)
  - Drag handle for reordering (optional — can be v2)
- "Add source" button at the bottom
- Each source row is compact — one line: `[label input] [url input] [delete btn]`
- Save changes on blur or with a save button (match existing pattern in the app)

### Frontend — Source status indicator (optional, nice to have)
- After session starts, show a small indicator next to each source:
  - Green dot: fetched OK
  - Red dot: fetch failed
- This requires the backend to report fetch status — could be part of the session health endpoint or a separate endpoint

### System prompt remains separate
- The system prompt field continues to work as before — user writes instructions there
- URL sources are a separate list, fetched and appended to context AFTER the system prompt
- Context order: system prompt first, then source content, then conversation history

## Implementation Notes
- For HTML content extraction, a simple approach is fine: strip all `<script>`, `<style>`, `<nav>`, `<footer>`, `<header>` tags, then extract inner text from the rest. No need for a full readability parser.
- Use Go's `net/http` client with timeouts for fetching
- Consider using `golang.org/x/net/html` for HTML parsing, or just regex-strip tags if simpler
- The context assembly in `context.go` already has a hierarchical structure — sources fit in as a new layer
- Existing tests must pass (`make check`)
- Write tests for the CRUD endpoints and the fetch/extract logic
