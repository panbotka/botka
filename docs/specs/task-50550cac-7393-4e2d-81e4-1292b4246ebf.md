# Preview uploaded files inline in chat

Show inline previews of uploaded files in chat messages — image thumbnails, PDF first-page preview, and download links for other file types.

## Requirements

### Backend — Serve uploaded files
- Add `GET /api/v1/uploads/:filename` endpoint that serves files from the upload directory
- Set appropriate `Content-Type` headers based on file extension/stored MIME type
- Basic security: only serve files that exist in the attachments DB table (prevent path traversal)

### Backend — Attachments in message response
- Ensure the message API response includes the `attachments` array with: `id`, `original_name`, `stored_name`, `mime_type`, `size`
- Add a `url` field computed as `/api/v1/uploads/<stored_name>`

### Frontend — Inline previews
- For messages with attachments, render previews below the message content:
  - **Images** (image/png, image/jpeg, image/gif, image/webp): show thumbnail (max ~300px wide), click to open full size in new tab or lightbox
  - **PDFs** (application/pdf): show a file card with PDF icon, filename, size, and "Open" link (opens in new tab via the serve endpoint)
  - **Text files** (text/plain): show a file card with text icon, filename, size
- Multiple files: display in a row/grid

### Frontend — Styling
- Thumbnails have rounded corners, subtle border
- File cards: icon + filename + size, compact design
- Follow existing app design language (Tailwind, zinc palette)
- Works in both streaming (live) and persisted message history

## Implementation Notes

- Attachment model already exists with `original_name`, `stored_name`, `mime_type`, `size`
- Messages already have `Attachments` relation via GORM `foreignKey:MessageID`
- Upload directory is configured via `UPLOAD_DIR` env var (default `./data/uploads`)
- Existing tests must pass (`make check`)

## Safety

**NEVER run `make deploy`, `make install-service`, `systemctl restart botka`, or any command that would restart the Botka service.**
