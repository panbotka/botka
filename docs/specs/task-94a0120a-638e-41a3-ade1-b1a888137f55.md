# Track and display cost per thread

Store cost and token usage from each assistant response and display cumulative cost per thread.

## Requirements

### Backend — Store cost on messages
- Add `cost_usd NUMERIC(10,6)` nullable column to `messages` table (migration)
- Add `input_tokens INT` and `output_tokens INT` nullable columns to `messages` table
- When saving an assistant message in `chat.go`, store the cost_usd, input_tokens, and output_tokens from the result event
- Update the Message model with `CostUSD *float64`, `InputTokens *int`, `OutputTokens *int` fields

### Backend — Thread cost endpoint
- Add `GET /api/v1/threads/:id/cost` endpoint
- Returns `{"data": {"total_cost_usd": 0.42, "total_input_tokens": 15000, "total_output_tokens": 8000, "message_count": 12}}`
- Aggregates from messages table: `SUM(cost_usd)`, `SUM(input_tokens)`, `SUM(output_tokens)`, `COUNT(*) WHERE role='assistant'`

### Backend — Include cost in thread list
- Add `total_cost_usd` to thread list response (computed via subquery or join)

### Frontend — Thread detail
- Show cumulative cost somewhere visible in the thread view (e.g. header area or info panel)
- Format: "$0.42" or "420 tokens in / 200 tokens out"
- Update live as new messages stream in (add cost from usage SSE event)

### Frontend — Thread list
- Show cost next to each thread in the thread list (subtle, muted text)
- Threads with zero/null cost show nothing

## Implementation Notes

- The `usage` SSE event already contains `cost_usd`, `input_tokens`, `output_tokens` — just need to persist them
- Message model already has `prompt_tokens` and `completion_tokens` — consider reusing or replacing with `input_tokens`/`output_tokens` for consistency with Claude's naming
- Existing tests must pass (`make check`)
