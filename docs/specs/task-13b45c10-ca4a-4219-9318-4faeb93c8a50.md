# Add started_at and completed_at timestamps to tasks

Track when a task started execution and when it finished. Sort done tasks by completion date (newest first) and show duration.

## Requirements

### Backend — Migration
- Add `started_at TIMESTAMPTZ` nullable column to `tasks` table
- Add `completed_at TIMESTAMPTZ` nullable column to `tasks` table
- Backfill existing tasks: for tasks with status done/failed/needs_review/cancelled that have executions, set `started_at` from the first execution's `started_at` and `completed_at` from the last execution's `finished_at`

### Backend — Model
- Add `StartedAt *time.Time` and `CompletedAt *time.Time` fields to `models.Task`
- Include both fields in JSON serialization (`started_at`, `completed_at`)

### Backend — Runner
- In `claimTask` (runner.go ~line 374): set `started_at` on the task when transitioning to `running` (only if `started_at` is currently nil, so retries don't overwrite)
- In `finalizeTask` (runner.go ~line 492): set `completed_at` on the task when writing final status (done/failed/needs_review)

### Backend — List endpoint sorting
- In `task.go` List handler (~line 121): when the status filter contains `done`, `failed`, `needs_review`, or `cancelled`, order by `completed_at DESC NULLS LAST` instead of `priority DESC, created_at ASC`

### Backend — Task list response
- Add `started_at` and `completed_at` to the task list item response so the frontend can compute duration

### Frontend — Types
- Add `started_at: string | null` and `completed_at: string | null` to the Task type

### Frontend — Task list (done tab)
- For tasks that have both `started_at` and `completed_at`, show duration (e.g. "2h 15m", "45m", "3m") in the task row
- Format should be human-friendly: use days/hours/minutes, skip zero components

## Implementation Notes

- Follow existing migration pattern in `migrations/` directory (next sequence number)
- The runner already tracks `started_at`/`finished_at` on `task_executions` — this adds it to the task level for simpler querying and display
- Backfill in the migration uses data from `task_executions` table
- Existing tests must pass after changes (`make check`)

## Safety

**NEVER run `make deploy`, `make install-service`, `systemctl restart botka`, or any command that would restart the Botka service.** Task agents run inside Botka — deploying would kill the agent's own process.
