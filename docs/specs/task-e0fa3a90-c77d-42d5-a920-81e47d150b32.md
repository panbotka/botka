## Task 16: Data Migration Script

Create a standalone tool to import existing data from the Saiduler and Chatovadlo databases into the new Botka database.

### Context
Both apps have existing data that should be preserved in the merged app. The Botka database uses a unified schema where projects replace both Saiduler's projects and Chatovadlo's folders.

### Source Databases
- **Saiduler:** `postgres://saiduler:saiduler@localhost:5432/saiduler` â€” projects, tasks, task_executions, runner_state
- **Chatovadlo:** `postgres://admin:...@localhost:5432/botka_chat` (check actual credentials in `/home/pi/projects/chatovadlo/.env`) â€” threads, messages, attachments, branch_selections, personas, tags, thread_tags, memories, folders

### Target Database
- **Botka:** `postgres://botka:botka@localhost:5432/botka`

### What to Create

**`cmd/migrate-data/main.go`** â€” Standalone Go program:

1. **Connect to all 3 databases** (source saiduler, source chatovadlo, target botka)

2. **Import from Saiduler:**
   - Copy `projects` rows â†’ target `projects` (preserve UUIDs)
   - Copy `tasks` rows â†’ target `tasks` (preserve UUIDs, project_id references)
   - Copy `task_executions` â†’ target `task_executions`
   - Copy `runner_state` â†’ target `runner_state` (upsert the singleton row)

3. **Import from Chatovadlo:**
   - Read `folders` table â†’ for each folder:
     - Try to match by `directory_path` to an existing project in target (by `path` column)
     - If match found: update the project's `claude_md` and `sort_order` from the folder
     - If no match: create a new project row from the folder data (generate UUID, use folder name/path, set active=false since it wasn't discovered by Saiduler)
     - Build a `folder_id â†’ project_id` mapping
   - Copy `personas` â†’ target `personas` (preserve IDs)
   - Copy `tags` â†’ target `tags` (preserve IDs)
   - Copy `threads` â†’ target `threads`:
     - Remap `folder_id` to `project_id` using the mapping built above
     - Preserve all other fields
   - Copy `messages` â†’ target `messages` (preserve IDs, parent_id references)
   - Copy `attachments` â†’ target `attachments`
   - Copy `branch_selections` â†’ target `branch_selections`
   - Copy `thread_tags` â†’ target `thread_tags`
   - Copy `memories` â†’ target `memories`

4. **Handle edge cases:**
   - Skip `passkey_credentials` (auth removed)
   - Handle NULL folder_id in threads (set project_id to NULL)
   - Handle duplicate project paths (folder path matches existing project)
   - Log progress and any skipped/failed rows
   - Use transactions for atomicity

5. **Make it idempotent** â€” can be run multiple times safely:
   - Use `ON CONFLICT DO NOTHING` or check existence before insert
   - Or truncate target tables first with a `--clean` flag

### Also

- Update the file paths or keep them as stored_name (UUID-based) which should be path-independent
