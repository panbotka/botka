## Problem

When a user sends a chat message, the Claude Code session briefly disappears from `/api/v1/processes` and then reappears. This is visible when polling the endpoint.

## Root Cause

In `internal/handlers/chat.go` `streamResponse()`, the sequence is:

1. `Pool.Acquire()` (pool.go:152-178) calls `Registry.Unregister(threadID)` — session **disappears** from `/api/v1/processes`
2. Then `streamResponse()` (chat.go:428) calls `Registry.Register(threadID, ...)` — session **reappears**

There's a brief window between Unregister and Register where the session is missing from the registry.

## Fix

Refactor the Acquire/Register sequence so the registry entry is never removed. Options:

1. **Don't unregister in Acquire()** — instead, have `streamResponse()` update/replace the existing registry entry atomically (e.g. `Registry.Replace()` method that swaps the cancel func without removing the entry).
2. **Move Register before Acquire** — register the new cancel context first, then acquire the pool entry. Since `Register()` already overwrites, this eliminates the gap.

Option 2 is simpler — just move line 428 (`Registry.Register(...)`) before the `Pool.Acquire()` call at line 436. The pool's `Acquire()` still calls `Unregister()`, but since `Register()` already ran with the new context, the entry is immediately present.

Actually the cleanest fix: remove the `Registry.Unregister(threadID)` call from `Pool.Acquire()` entirely (pool.go:166). The `Registry.Register()` in `streamResponse()` already overwrites the entry, so there's no need to unregister first.
