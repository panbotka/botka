# Signal Bridge Background Service

Background service that bridges messages bidirectionally between Signal group chats and Botka threads. When a message arrives in a linked Signal group, it appears in the Botka thread and triggers a Claude auto-reply. When Claude responds, the response is sent back to Signal. Messages sent from Botka UI are also forwarded to Signal.

## Requirements

### Incoming messages (Signal → Botka)

- Create a new package `internal/signal/bridge.go` (or similar) with a `Bridge` struct that runs as a background goroutine
- Start the bridge when the Botka server starts (alongside the task runner)
- Poll signal-cli for incoming messages at a regular interval (every 2-3 seconds)
- Use the `signal.Client.Receive()` method from the signal client package (`internal/signal/`)
- NOTE: The signal-cli daemon may be in auto-receive mode, which causes `receive` to fail. If that happens, investigate alternative approaches — check if the daemon exposes a subscription/streaming mechanism, or whether it needs to be restarted with `--receive-mode manual`. The daemon runs at `127.0.0.1:5107` and can be inspected.

For each incoming group message:
1. Look up `signal_bridges` table to find a thread linked to this group ID
2. If no bridge exists for this group, skip the message
3. Create a `Message` record in the linked thread with:
   - `role`: `"user"`
   - `content`: the message text, prefixed with sender name like `**Tomáš:** Hello` so it's clear who wrote it in the Botka UI
   - Set appropriate `parent_id` (latest message in the thread's active conversation path)
4. Trigger a Claude auto-reply by programmatically invoking the same logic used by the chat handler (`internal/handlers/chat.go` `SendMessage`). Study how `SendMessage` and `streamResponse` work — you need to:
   - Assemble context (same as regular chat)
   - Spawn Claude Code subprocess (same runner)
   - Collect the full response (you don't need SSE streaming for Signal — just collect the complete text)
   - Save the assistant message to the thread
5. Send Claude's response text back to the Signal group via `signal.Client.SendGroupMessage()`

### Outgoing messages (Botka → Signal)

When a message is sent from the Botka UI to a thread that has an active Signal bridge:
- After the assistant (Claude) message is saved, also send it to the linked Signal group
- The user message typed in Botka UI should also be forwarded to Signal (so Signal participants see both sides)
- Hook into the existing chat handler flow — after assistant message persistence, check if the thread has a signal bridge and forward if so

### Bridge lifecycle

- Bridge goroutine starts with the server
- Graceful shutdown on server stop (context cancellation)
- If signal-cli is unavailable, log warning and retry on next poll cycle (don't crash)
- Skip messages that are from Pan Botka's own number (+420723750652) to avoid reply loops

### Duplicate prevention

- Track the last processed message timestamp per group to avoid reprocessing
- Signal message timestamps (unix milliseconds) are unique identifiers

### Edge cases

- If Claude is already responding in this thread (from Botka UI), queue the Signal message and process after current response finishes
- If the bridge is set to `active: false`, skip message processing for that bridge
- Handle empty/media-only messages gracefully (skip or note "sent an attachment")
