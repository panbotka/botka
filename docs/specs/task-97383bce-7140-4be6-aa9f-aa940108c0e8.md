# Remote Claude Code Execution via SSH on Box

Enable Botka to spawn Claude Code sessions on a remote build machine (Box) via SSH, so compute-heavy projects can run on powerful hardware instead of the Raspberry Pi.

## Background

Botka runs on a Raspberry Pi. A powerful x86_64 build machine ("Box") is available on the Tailscale network at `box@box` (AMD 12-core, 64GB RAM, RTX 3070). Box has Claude Code CLI installed and ready. Box is not always on — it must be woken via Wake-on-LAN (`boxon` command) and takes ~30 seconds to boot.

Currently, Claude Code subprocesses are spawned locally. This task adds the ability to spawn them remotely via SSH.

## Requirements

### Remote execution in Claude runner

Modify `internal/claude/runner.go` to support remote execution:

- When a thread's working directory points to a remote path, spawn Claude Code via SSH instead of locally
- Remote command: `ssh box@box claude -p "..." --resume <session-id> ...` (same args as local, just prefixed with SSH)
- SSH must pass through stdin/stdout for NDJSON streaming — this works natively with `ssh` as it forwards stdio
- The remote working directory is set via `--project-dir` flag or `cd` before Claude invocation
- SSH connection should use the existing key-based auth (no password needed, already configured)

### Remote directory convention

- Remote paths are identified by a prefix convention: `box:` prefix (e.g., `box:/home/box/projects/myapp`)
- When the user selects a working directory for a thread, they can choose local paths (`/home/pi/projects/...`) or remote paths (`box:/home/box/projects/...`)
- The `box:` prefix is stripped when constructing the SSH command — the actual path after prefix is the working directory on Box

### Wake-on-LAN integration

Before establishing an SSH connection to Box:
1. Try a quick SSH connectivity check (e.g., `ssh -o ConnectTimeout=3 box@box true`)
2. If it fails, run the WoL command (configured via `BOX_WOL_COMMAND` env var, default `/home/pi/bin/boxon`)
3. Poll SSH connectivity every 5 seconds, up to 60 seconds timeout
4. If Box doesn't come up within timeout, fail the operation with a clear error message
5. Cache the "Box is alive" state for 5 minutes to avoid repeated WoL checks

### Configuration

Use existing env vars from config:
- `BOX_HOST` (default `100.127.79.1`) — but for SSH use `box@box` hostname which resolves via Tailscale
- `BOX_SSH_USER` (default `box`)
- `BOX_WOL_COMMAND` (default `/home/pi/bin/boxon`)

Add new env var:
- `BOX_SSH_HOST` (default `box`) — SSH hostname for Box

### Task executor support

Also modify `internal/runner/executor.go` so task execution supports remote projects. If a task's project has a remote path, the executor should use the same SSH-based Claude spawning.

### Session management

- Session IDs are managed by Botka (stored in thread records) as before
- Session files physically live on Box's filesystem — Botka doesn't need to access them directly
- `--resume <session-id>` is passed via the SSH command
- Session validation (`SessionExists()`) needs to work remotely — either skip validation for remote sessions, or run a quick SSH check

### Edge cases

- SSH connection drops mid-session: detect the broken pipe, report error, allow retry
- Box goes to sleep during a session: same as connection drop
- Multiple concurrent remote sessions: SSH handles this fine, no special logic needed
- Local and remote sessions can run simultaneously (different threads)
